# WOM 설계 철학 구현 근거

상태: v0.3.255에서 다시 검토한 v0.3.252 공개 추적성 점검
날짜: 2026-07-16

## 목적

WOM의 설계 철학은 설득력 있는 글에만 머물면 안 됩니다. 이 문서는 핵심
원칙마다 실제로 적용하는 실행 표면, 그 원칙을 지키는 회귀 테스트, 그리고
WOM이 아직 정직하게 증명할 수 없는 경계를 연결합니다.

근거는 의도적으로 세 층으로 나눕니다.

1. **공학적 구현**: 공개 릴리스에 명령, 스키마, 영수증, AI 운영 규칙 또는
   결정론적 검사가 있습니다.
2. **실사용 검증**: 실제 아카이브에서 그 장치가 유용한 의미를 보존하는지는
   사람과 운영 AI가 계속 시험해야 합니다.
3. **외부 서비스별 미래 작업**: 원격 작업이나 완료 상태를 주장하려면 해당
   외부 서비스에 맞는 별도 계약이 필요합니다.

공학 검사를 통과했다고 다른 두 층까지 자동으로 충족되는 것은 아닙니다.

## 추적성 표

| 아카이브는 지속되는 기억이고, 채팅은 임시 작업 기억입니다. | 배포되는 Agent Skill은 모든 아카이브 세션을 `ai-start-here`에서 시작합니다. `ops/operational-context.yml`, `ai-artifact-inventory`, `session-handoff-checkpoint`가 영수증 기반 인수인계 경계를 만듭니다. | `test_runtime_skill.py`, `test_runtime_skill_install.py`, `test_cli.py`의 세션 인수인계 사례. | WOM은 호스트 채팅을 직접 읽거나 의미의 완전성을 증명할 수 없습니다. 사람 또는 호스트 AI가 대화를 명시적으로 검토하고 중요한 내용을 기록해야 합니다. |

| `정본`은 사람이 검토한 현재 아카이브 상태이지 객관적 진실이 아닙니다. | AI 운영 지침과 수정 흐름이 사람 검토, 정확한 해시, 승인, 영수증, 복구 가능한 이전 바이트를 요구합니다. | 아티팩트 우선 문서 테스트와 정본 수정 계획·쓰기·감사·복원 테스트. | 영수증은 제한된 작업과 근거를 증명할 뿐, 영원한 진실이나 보편적 동의를 증명하지 않습니다. |
| 읽기는 선언된 범위의 모든 zet를 빠짐없이 세고, 초록 데이터는 그 읽기 순서에만 사용합니다. | `first-read-readiness`, 엄격한 페이지형 `zet-catalog`, 스냅샷과 이어읽기 검사, 토큰 예산, MCP 이어읽기, `zet-catalog-pass`가 생성 지도를 정본으로 만들지 않고 선택된 모든 zet를 셉니다. | `test_cli.py`, `test_mcp_server.py`, `test_zet_catalog_benchmark.py`의 카탈로그와 첫 읽기 사례. | 구조적 완전성이 초록의 의미 품질을 증명하지는 않습니다. 초록은 읽는 순서를 정하며, 사람의 목표에 필요하면 완전한 zet 본문과 출처 근거를 계속 읽어야 합니다. 공개 언어의 `node`는 주체/archive 참여자를 뜻하며, 그래프 코드에서만 내부 용어로 `zet vertex`를 쓸 수 있습니다. |

WOM은 지속되는 맥락과 근거를 기록하지만 호스트 세션 생명주기를 통제한다고 주장하지 않습니다. |
| 타이, 엣지, 인덱스, 임베딩, 그래프는 통로나 검토 가능한 주장이지 권위가 아닙니다. | AI 운영 지침이 조용한 동일성 병합을 금지합니다. 엄격한 카탈로그 순회는 실제 노드를 기준으로 하며, 전체 카탈로그 산출물은 SHA로 묶어 읽고 승인 뒤 정리하는 비공개 임시 파일입니다. | runtime skill과 아티팩트 우선 문서 검사, 그리고 실제 카탈로그 통과·읽기·정리 동작 검사. | WOM에는 전역 엔티티 해석기가 없으며, 미래의 모든 외부 연동이 이 경계를 지킬 것이라고 주장하지 않습니다. |
| 사람이 검토한 로컬 상태가 권위이고, 원격 시스템은 백업 또는 복제 계층입니다. | `local-sovereignty`가 권위 모델을 선언합니다. `backup-evidence`는 설정을 완료 증거로 바꾸지 않고 GitHub, 오브제 저장소, 외부 DB 영역을 따로 보고합니다. | `test_cli.py`의 로컬 주권·백업 근거 사례와 기능표·문서 검사. | GitHub나 외부 DB의 범용 완료 영수증은 없습니다. 오브제 영수증은 기록 시점의 범위만 증명하며, 지금 원격에 존재하는지 또는 전체 백업이 끝났는지는 증명하지 않습니다. |

| AI 생성 문서와 대화에서 나온 작업은 증발하면 안 됩니다. | `ai-artifact-inventory`가 로컬 AI 산출물을 분류하고, 작업기록이 미완료 작업을 남기며, `session-handoff-checkpoint`는 지속 기록 근거가 없거나 낡으면 깨끗한 인수인계를 막습니다. | AI 산출물 목록과 세션 인수인계 CLI 테스트. | 도구는 채팅을 자동 수집하거나 어떤 생성 문서를 보존해야 하는지 대신 결정하지 않습니다. 사람과 AI의 검토가 계속 필요합니다. |
| AI 운영은 필요한 것부터 단계적으로 읽고 사람에게 쉬운 말로 설명해야 합니다. | 배포되는 Agent Skill은 짧은 시작 문서, 목표별 참고문서, 기계 판독형 기능 목록, 사람말 응답 계약과 용어 안내를 제공합니다. | runtime skill 패키지 검사, 기능표 테스트, 문서 계약 테스트. | 쉬운 설명과 좋은 판단은 AI 지침 수준의 동작입니다. WOM은 모든 모델 답변의 품질을 결정론적으로 검사할 수 없습니다. |

## 현재 공학적 결론

공개 v0.3.x 구현에는 메멘토 문제를 다루는 구체적이고 회귀 테스트된 장치가
있습니다. 첫 읽기 재구성, 아티팩트 우선 추론, 검토된 수정과 복구, 지속되는
세션 인수인계, 정직한 로컬 백업 근거는 단순한 로드맵 문구가 아닙니다.

그렇다고 WOM이 완성됐다는 증거는 아닙니다. 남은 경계는 주로 실증 또는 외부
서비스별 작업입니다.

- 실제 말뭉치가 변할 때 초록 데이터가 의미상 계속 유용한지 시험해야 합니다.
- 세션 인수인계가 사람이 정말 중요하게 여긴 결정을 보존하는지 시험해야 합니다.
- 수정과 복원 흐름은 실제 사용 관찰을 계속 받아야 합니다.
- GitHub와 외부 DB 완료를 주장하려면 외부 서비스별 근거 계약이 필요합니다.
- 지금 원격 오브제가 존재하는지 확인하려면 로컬 영수증만이 아니라 실제 외부
  서비스 검증이 필요합니다.

이것들은 숨겨진 구현 주장이 아니라 명시된 검증 경계입니다.
