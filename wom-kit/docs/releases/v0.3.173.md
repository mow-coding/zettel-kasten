# v0.3.173 - base-link-types append-only no-clobber sync

v0.3.173 lands one additive command. It needs no migration, and leaves every default
path byte-identical to v0.3.172.

## What ships — `archive migrate --target base-link-types`

`archive migrate <archive-root> --target base-link-types (--dry-run | --approve)
[--reviewed-by <actor>] [--format json]` is an append-only, no-clobber sync that pulls
every base WOM-kit link type missing from a stale archive-local `zettel-kasten/types.yml`
into that file.

- **Append-only, no-clobber.** Missing base entries are deep-copied verbatim (full entry:
  id, description, from/to, and any policy notes) and appended AFTER the archive's
  existing `link_types`. No existing entry is removed, renamed, reordered relative to
  other existing entries, or altered in value. An archive that customized a base id with a
  divergent value keeps its own entry — that id is reported under `present_not_overwritten`
  and never rewritten.
- **Superset of the recommended-9.** Its id set is a strict superset of the
  `link-types-v0.3` recommended connection-edge set (`material`, `derived`, `semantic`,
  `embed`, `mention`, `contains`, `supersedes`, `view_query`, `comment_context`). It also
  pulls `continues` (added to the base in v0.3.168) and any other base-only id. After a
  sync, `migrate --target link-types-v0.3` is a no-op; after a prior `link-types-v0.3`
  migrate, sync adds only the non-recommended remainder (including `continues`).
- **`--reviewed-by` is required with `--approve`.** This matches the house convention for
  approval-gated writes. Because the sync receipt records `reviewed_by`, it is a distinct
  receipt variant from the reused v0.1 migration receipt.
- **Distinct receipt namespace.** On approve it writes a receipt with
  `receipt_kind: base_link_types_sync` and
  `schema_version: wom-kit/base-link-types-sync-receipt/v0.1` under
  `receipts/migrations/base-link-types.<digest>.migration.json`. The distinct target
  segment and `receipt_kind` guarantee a `link-types-v0.3 --revert` can never pick up a
  sync receipt (revert filters on `receipt_kind == "migration_write"` and further
  intersects candidates with the recommended-9 set).
- **Atomic with rollback.** The types.yml write and the receipt write are wrapped so a
  failure rolls types.yml back to its original bytes, unlinks any partial receipt, and
  re-raises — no orphan `.tmp` or receipt remains.
- **Idempotent.** A second run with nothing missing is a clean no-op: no receipt is
  computed, no receipt-exists blocker is reachable, `files_written == []`, `ok == True`,
  `blocked == False`.

## No-`types.yml` safe branch

If the archive has **no** local `zettel-kasten/types.yml`, sync writes nothing and creates
no file. Such an archive transparently inherits ALL current and future base link types via
the base fallback in `load_allowed_link_types`. Creating a local `types.yml` would
permanently shadow (freeze) that inheritance at today's base — a silent, irreversible
regression directly opposed to the command's purpose. The result reports
`inherits_base_directly: true` and `archive inherits base directly; nothing to sync`.

## No revert

There is deliberately **no `--revert`** for `base-link-types`. A symmetric revert would
have to distinguish sync-appended ids from owner-authored ids and could remove `continues`
etc. that the owner now relies on. `--revert --target base-link-types` fails closed with
`base-link-types migration does not support --revert.` The recommended-set
`link-types-v0.3` revert is unchanged.

## Discoverability

`archive doctor` now appends a remedy hint to its `zettel_edge_type_unknown` warning: an
operator hitting an edge type undefined in the archive's `types.yml` is routed to
`archive migrate --target base-link-types --dry-run` to preview pulling missing base link
types. No blocker or exit semantics change.

## Honesty contract

1. Once an archive has its **own** `zettel-kasten/types.yml`, it shadows the base
   permanently — every future base link type also needs a manual `migrate --target
   base-link-types`. There is no automatic propagation.
2. Sync copies base entry shapes **as of this release** (a snapshot, not a live link).
3. Sync **normalizes/rewrites the whole `types.yml`** via `safe_dump` (comments, anchors,
   flow-style, and key ordering may be normalized) — exactly like the sibling
   `link-types-v0.3` migration. Existing link-type entries are preserved by value/id;
   surrounding formatting is not byte-preserved. Review the diff.
4. `continues` (added to the base v0.3.168) is the concrete case this command exists to
   pull into archives that already vendored a `types.yml`.

## Compatibility

Additive command; **no migration required**. The default paths are byte-identical to
v0.3.172. The two pinned `link-types-v0.3` tests, the capability-matrix doc-test, and
`check_public_privacy` / `check_release_readiness` all stay green.
