# v0.3.246 - Artifact Primacy And Human Drift

v0.3.246 records a core WOM boundary: the project may borrow auditable
operating discipline from enterprise ontology systems, but it must not turn a
human archive into one supposedly final entity graph.

## What Changed

- Added the artifact primacy and human drift doctrine to the English and
  Korean product philosophy.
- Extended the foundational whitepaper and product blueprint with the same
  authority model and implementation consequences.
- Clarified that a `canonical` zet is the subject-approved current archive
  state, not an objective-truth certificate.
- Declared contradictions, ambiguity, repetition, correction, and changing
  meanings to be potentially valuable chronological evidence.
- Declared nodes, ties, edges, embeddings, indexes, entity candidates, and
  graph projections to be regenerable reading aids or reviewable claims beneath
  local artifact authority.
- Prohibited silent identity merges based only on matching labels.
- Added a public decision log, README visibility, capability-matrix status, and
  focused documentation regression tests.

## Why It Matters

WOM's primary subject is a human, not a stable master-data table. The same word
or name can carry a different meaning at another time. A graph that erases
those differences can look coherent while telling a false story.

AI may use a different token budget or a stronger future model to reconstruct
context from the archive. That inference can change. The durable layer is the
time-situated artifact, its provenance, and the history of how the person
revised or contradicted it. WOM's goal is not deterministic knowledge. It is to
help a person notice how their own knowledge changes over time.

## Safety Boundary

- No archive format or schema changes.
- No migration or rewrite of existing zets.
- No entity resolver, automatic merge, model call, embedding generation, or
  graph mutation.
- No new CLI, MCP, provider, database, object-store, host-configuration, or UI
  authority.
- Existing entity-term quality checks remain warning/blocker policy tools; they
  do not establish a global entity truth layer.

## Verification

Release verification covers focused doctrine regression tests, the full
WOM-kit suite, deterministic package resources, clean-wheel installation,
public link hygiene, Korean product-language hygiene, public privacy hygiene,
and release readiness.
