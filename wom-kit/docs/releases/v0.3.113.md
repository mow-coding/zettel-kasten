# v0.3.113 - Account Recovery Break-Glass Scenario

Date: 2026-06-19

## What Changed

This release handles a small but important security feedback item after
v0.3.112: where WOM should point humans when they need to store 2FA recovery
codes, backup codes, or other emergency-only access material.

`credential-store-recommendation` now supports two new scenarios:

- `account_recovery_codes`
- `break_glass_secrets`

Both scenarios keep WOM out of the vault. WOM recommends safe storage policy and
metadata only; it does not ask for, read, store, or echo recovery code values.

## Break-Glass Storage Guidance

The new account recovery scenario recommends:

- one encrypted digital copy in a KeePassXC-style `secret:` entry,
- one independent offline copy, represented only by a safe label such as
  `offline:physical-safe`,
- at least two independent locations,
- no single digital-only copy,
- an explicit circular-dependency check before relying on the vault.

The circular-dependency check matters when the protected account controls vault
sync, device login, email recovery, or password-manager access. In that case,
the vault copy alone cannot be the only recovery path.

## Semantic Routing

`credential-semantic-extraction-recipe` now reports store-scenario hints for the
highest-risk semantic classes:

- `recovery_codes -> account_recovery_codes`
- `wallet_seed_or_private_key_material -> break_glass_secrets`

Wallet seed and private-key material still requires separate custody review and
is not treated as a generic credential migration item.

## Safety Boundary

This release does not:

- read recovery code files,
- detect secret values,
- open KeePassXC or any other password manager,
- open OS keyrings,
- read browser password stores,
- write vault entries,
- store recovery code values in WOM,
- echo recovery code values into JSON, logs, docs, prompts, or receipts,
- add secret retrieval adapters,
- add provider calls,
- add MCP write tools.
