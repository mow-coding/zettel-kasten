# v0.3.209 - honest first-read readiness

v0.3.209 prevents an AI host from confusing “every zet file was visited” with
“every zet had a readable abstract.” It is additive and requires no archive
migration.

## What Changed

- Kept `archive_wide_coverage_claim_ready` as strict, unskipped file-node
  visitation evidence.
- Added `archive_wide_abstract_reading_claim_ready` for complete non-redacted
  first-read availability.
- Added `archive_wide_followup_resolution_ready` for safe unique id follow-up.
- Added `abstract_coverage` and `identity_coverage` count summaries.
- Updated runtime/start-here/MCP/agent guidance so hosts report gaps instead of
  inventing abstracts or treating duplicate ids as safely resolvable.
- Extended the synthetic catalog benchmark to assert all three readiness
  signals on its reviewed, unique-id fixture.

## Compatibility And Safety

Existing catalog inputs, path/seeded order, page budgets, and continuation
tokens remain compatible. The catalog schema advances to v0.5; continuation
schema remains v0.2. Existing zets are not rewritten.

Redacted nodes still count toward file-node coverage but are excluded from the
abstract requirement. Missing/unreadable first-read text and duplicate ids are
reported, not repaired. No body, objet byte, secret, provider, generated index,
or persistent WOM goal/loop state is read or created.
