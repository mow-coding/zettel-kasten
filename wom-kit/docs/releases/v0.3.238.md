# v0.3.238 - Chronological Revision Event Chain

v0.3.238 makes canonical revision history safe for future exact-byte restore
events while keeping the entire release read-only.

## What Changed

- Replaced unique-state backward traversal with per-canonical chronological
  event ordering.
- Required unique normalized event times and exact adjacent file, semantic,
  abstract, and body before/after evidence.
- Allowed evidence-complete repeated states such as `A -> B -> A` without
  weakening branch, replay, evidence-gap, identity, or current-tip checks.
- Required `zet-revision-restore-plan` to bind the actual latest receipt event.
  Matching current bytes to an older repeated state is no longer enough.
- Bound latest-event digest and receipt SHA evidence into the restore plan
  digest without echoing private identity, paths, or text.

## Upgrade And Compatibility

No archive migration is required. Restart the AI operator process and rerun
`zet-revision-receipt-audit --dry-run` before preparing any restore. Existing
v0.3.235 revision receipts remain valid.

The audit complexity is now
`O(receipt_files log receipt_files + revision_chains + lock_files)` because
events must be ordered. It still opens each receipt and lock once, opens each
current canonical target at most once per private identity, and performs no
quadratic or per-receipt whole-archive scan.

## Safety And Claim Boundary

- A repeated hash is allowed only when adjacent full-state evidence and event
  times form one continuous history.
- Duplicate event times, branch/replay transitions, partial state gaps, and
  current-tip drift block.
- The selected restore receipt must be the actual newest event even when an
  older receipt's after-state bytes match current bytes again.
- No canonical, receipt, lock, object, provider, or database state is written.
- No restore writer or duplicate MCP tool exists in this release.
