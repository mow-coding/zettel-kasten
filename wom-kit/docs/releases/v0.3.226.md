# v0.3.226 - Archive Identity Consistency And Reviewed Repair

v0.3.226 makes duplicated archive identity metadata explicit and repairable
without silently rewriting a real archive. It is additive and requires no
automatic migration.

## What Changed

- Quick runtime-context and AI start-here report `identity_consistency` without
  scanning zets, objets, or receipt corpora.
- Doctor reports archive-id, scope, principal-id, display-name, missing-id, and
  template-residue findings with a read-only repair route where safe.
- `archive.yml principal` is the archive principal declaration;
  `archive-identity.yml` remains the identity and ownership core.
- New `init` and onboarding archives replace template identity id/display
  metadata and begin in the aligned state.
- `archive identity-reconcile <archive-root> --dry-run --format json` previews
  only changed field names and current/proposed SHA-256 digests.
- Approval requires an attributed reviewer, principal-metadata affirmation,
  and the exact archive, current-identity, and proposed-identity digests.
- Successful approval edits only `archive-identity.yml`, verifies the result,
  and writes a schema-backed value-free receipt.
- Principal/archive conflicts and non-repairable findings fail closed.
- A handled receipt-write failure restores the exact original identity bytes.
- `VERSIONING.md` now reports the current public and package version instead of
  the historical v0.3.1 showcase checkpoint.

## Operator Flow

```text
archive ai-start-here <archive-root> --dry-run --format json
archive identity-reconcile <archive-root> --dry-run --format json
```

If the second command reports `repair_ready`, inspect `archive.yml` and
`archive-identity.yml`, then use the complete `approval_command` returned by
that same dry-run. Do not shorten it or reuse it after either file changes.

After an approved repair:

```text
archive runtime-context <archive-root> --strict --format json
archive doctor <archive-root> --strict
```

## Privacy And Work Boundary

Dry-run output and receipts contain no display-name or identity values. The
command reads only the two archive control files, calls no provider, reads no
credential store, and does not read zets or objets. Approval never edits
`archive.yml`.

Approval reserializes YAML. Parsed semantic changes are limited to the listed
fields, but comments, anchors, flow style, or formatting may normalize. The
exact original bytes are restored on handled in-process failure; forced process
or machine termination remains outside that rollback guarantee.
