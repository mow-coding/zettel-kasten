# v0.3.177 - force-reupload ledger-only hardening

v0.3.177 hardens the live object-storage `--force-reupload` path after the basoon
v0.3.176 verification letter showed a force run returning `skipped_already_present`
with **0 provider PUTs**. It is additive: no migration, no default-path behavior change, and
existing receipts/manifests remain valid.

## The problem this fixes

v0.3.175 introduced `object-storage-upload --force-reupload` so an operator could deliberately
RE-PUT an already-present object and prove a live provider PUT or a forced small multipart.
That path correctly bypassed the manifest-level present+match skip when the plan row was
`already_uploaded`.

The field result exposed a second skip authority: the per-object resume ledger. A post-crash or
handoff state can leave a terminal resume-ledger row even when the manifest no longer has a
`wom_uploaded` location for that object. In that ledger-only state the plan row is not
`already_uploaded`, so v0.3.176 did not set the executor's `force_upload` flag. The executor then
honored the terminal ledger row and returned:

```text
result_status: skipped_already_present
put_calls: 0
part_count: 0
provider_api_called: false
```

That is wrong for a reviewed `--force-reupload`: the operator explicitly asked for a live PUT
proof.

## The fix

`object_storage_upload_run` now passes `force_upload=True` into the executor whenever
`force_reupload` is active, not only when the manifest re-HEAD branch proved absence or
present+match. This means the force decision outranks the resume ledger in both common states:

- manifest says `already_uploaded` and the live re-HEAD would otherwise skip;
- manifest lacks a `wom_uploaded` location, but the resume ledger still has a terminal success row.

The existing safety invariants are preserved:

- `--force-reupload` still requires `--approve` and `--reviewed-by`;
- non-sha-derived key strategies are still refused;
- `--dry-run` is still inert/refused;
- local bytes are still hashed before any PUT, and a corrupt local file is refused before upload;
- HEAD-after re-download-and-hash verification, orphan cleanup, multipart part-count reporting,
  and the cumulative PUT ceiling still apply.

## Fail-closed output

Force execution results now include `forced_reupload: true` in the run output as well as in the
execution receipt. If a force run reaches a result with `put_calls == 0`, the run adds
`force_reupload_not_performed`, reports `ok:false`, and does not pretend the live proof happened.
This covers both the repaired ledger-only skip and future regressions that might reintroduce a
zero-PUT force path.

## Tests

The force-reupload regression suite now covers:

- the normal present+match force re-PUT path;
- forced small multipart with `part_count > 1`;
- default no-flag idempotent skip;
- corrupt local bytes refused before PUT;
- HEAD-after mismatch orphan cleanup;
- approval/reviewer/key-strategy gates;
- receipt privacy/validator compatibility;
- PUT ceiling enforcement;
- the new ledger-only post-crash/handoff state where a terminal resume ledger exists but the
  manifest lacks `wom_uploaded`;
- the new fail-closed guard for any future forced zero-PUT result.

## Compatibility

Additive; **no migration required**. Existing archives, manifests, and receipts are unaffected.
Operators who do not pass `--force-reupload` get the same idempotency behavior as before.
