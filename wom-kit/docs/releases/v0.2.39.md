# Release v0.2.39

Date: 2026-05-25

## Summary

v0.2.39 adds a CLI-only approved write layer for foreign block attestation review candidates.

The new command records that a human/operator approved an eligible quarantined foreign block as an untrusted attestation review candidate. It still does not create an attestation, signature, trust, import, mint, share, provider call, ZET transport, or acceptance.

## Added

```bash
archive record-attestation-review-candidate <archive-root> --candidate-plan <json-file> --dry-run --format json
archive record-attestation-review-candidate <archive-root> --candidate-plan <json-file> --approve --reviewed-by <actor-id> --format json
```

Optional replay guards:

```bash
--expected-case-id <case-id>
--expected-review-scope <scope>
--expected-attestor <actor-id>
--review-note <short-safe-note>
```

MCP adds read-only:

```text
record_attestation_review_candidate_check
```

## Compatibility

No private archive migration is required.

Approved CLI mode writes exactly:

```text
quarantine/foreign-blocks/<case-id>/attestation-review-candidate.json
receipts/quarantine/<case-id>.foreign-block-attestation-review-candidate.json
```

The write refuses overwrites and rolls back partial candidate files if the matching receipt cannot be written.

## Safety

- Candidate plans are validated as untrusted input.
- Current quarantine case, original quarantine receipt, decision record, and decision receipt are re-read before dry-run or approve output.
- Approved records remain `untrusted_foreign`, `recorded_untrusted_candidate`, and `not_created`.
- Raw review notes and original foreign body text are not copied into the candidate record or receipt.
- MCP remains read-only and cannot approve or write candidates.

## Non-Goals

v0.2.39 does not implement real foreign block trust, import, acceptance, attestation creation, signature creation, minting, sharing, provider API calls, ZET transport, payment, staking, consensus, blockchain behavior, OCR, LLM classification, or full-auto execution.
