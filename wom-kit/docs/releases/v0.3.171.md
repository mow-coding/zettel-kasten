# v0.3.171 - object-storage-adopt-existing --key-map (the 158 GB false-skip fix, hardened)

v0.3.171 adds one opt-in flag, `--key-map`, to `object-storage-adopt-existing`. It
lets an operator hand WOM the EXACT existing remote key for each object, so objects
a client already stores in their bucket under their own per-object key layout are
ADOPTED (and therefore not re-uploaded) instead of being re-PUT from scratch. The
release is additive and adopt-only: the default path is byte-identical to v0.3.170,
and `object-storage-upload` is untouched.

## The problem this fixes

A client's ~158 GB / ~19k objects already live in R2 under their own scheme —
`archives/archive:personal:basoon/objets/<sha>.<ext>` — with the ORIGINAL per-object
filename extension. For objects ingested through the prehashed external ledger, the
manifest `logical_key` is a bare content-addressed path with NO filename extension
(`objects/external/prehashed/<store_kind>/<first2>/<digest>`). So
`_object_storage_recovered_extension` returns `None`, `--key-append-extension` is a
no-op, the resolved template key is `<prefix>/<sha>` (no extension), the live HEAD
404s, and 0 of 19k objects adopt. The extension exists only in the client's own
upload records, which WOM has never seen. `--key-map` bypasses the broken template
by taking the exact key from the operator directly.

## What ships

- **`--key-map <file>` on `object-storage-adopt-existing` (adopt subcommand only).**
  A JSONL file, UTF-8, one object per line, exactly:

  ```
  {"sha256":"<64 lowercase hex>","remote_key":"<exact bucket-relative key>"}
  ```

  For each mapped object the map value becomes the resolved `remote_key` verbatim —
  the `object_storage_remote_key` template is NOT called, and
  `--key-strategy`/`--key-prefix`/`--key-append-extension` are IGNORED for that
  object. Objects with no map entry are reported and NOT adopted (they do not fall
  back to the template).

- **A stronger digest binding for operator-supplied keys.**
  `object_storage_map_key_binds_digest_segment` requires the object's 64-hex sha256
  to appear as a full `/`-delimited path segment OR as the filename stem (the tail
  after the last `/`, up to its first `.`). This is strictly stronger than the
  shipped `digest in remote_key` substring check. A map that points object A's sha at
  object B's real key is refused per-row (that object re-uploads), because B's key
  contains no A segment or stem.

- **Size stays manifest-sourced; the map carries no trusted size.** `size_bytes`
  always comes from the manifest record. A mapped object whose manifest size is null
  never adopts on presence alone — it is counted separately
  (`mapped_but_no_manifest_size_count`). A mapped key that 404s or size-mismatches at
  the live HEAD re-uploads; it never false-skips.

- **A fail-closed map loader.** `read_object_storage_key_map` mirrors the repo's
  prehashed-ledger discipline: `open("r", encoding="utf-8")`, per-line `json.loads`,
  `except (OSError, UnicodeError)` → a blocker; a leading UTF-8 BOM on the first line
  is stripped. The WHOLE file is parsed and validated before any adopt write. Any
  whole-run-fatal condition (unreadable/non-JSONL, a row missing a field, a non-hex
  sha, an unsafe `remote_key`, a duplicate sha with conflicting keys, or more than
  200,000 entries) appends a blocker and adopts ZERO — never a partial adopt from a
  half-parsed map. A duplicate sha with the identical key is deduped.

- **Reporting.** `adopt_summary` gains `key_map_used`, `mapped_count`,
  `unmapped_count`, `map_rejected_count`, and `mapped_but_no_manifest_size_count`;
  each `adopt_results` row carries `key_source` ∈
  {`template`,`map`,`map_rejected`,`unmapped`}. When a template run has
  `--key-append-extension` on and the extension is unrecoverable for a high fraction
  of objects (the prehashed signature), a discoverability warning names `--key-map`
  and the runbook.

## Data-safety: the residual gap, stated plainly

Presence+size adoption with `--key-map` trusts the operator's key→bytes mapping for
content identity. The digest binding proves the key is addressed to that object's own
slot, and the live HEAD proves an object of the same length is present at that key —
but a HEAD proves same LENGTH, not same BYTES. A hostile or hand-edited map whose key
binds the digest AND points at a same-size but different object would false-adopt.
`--content-hash-verify` is the ONLY cryptographic proof: it downloads and re-hashes
the object, and a hash mismatch does not adopt. For any map that is not mechanically
generated from a trustworthy per-object upload record, run with
`--content-hash-verify`.

## Mime-derived extension: a lossy fallback, not the default (acknowledged)

The stored manifest carries a `mime`, and `extension_from_mime_hint` maps common
mimes to an extension (`application/pdf` → `.pdf`, `application/x-hwp` /
`application/haansofthwp` → `.hwp`, `image/*` → `.png`, and so on). This is a LOSSY
recovery path: it cannot distinguish `.jpg` from `.jpeg`, and `application/octet-
stream` yields nothing, so a mime-built key can still HEAD-404. WOM does NOT use it to
build adopt keys automatically; it is mentioned only as a fallback in the
discoverability hint. The exact-key `--key-map` is the reliable path.

## Why adopt-only

`object-storage-upload` is Stage-2-gated and fails closed in this release; adding a
map to the live-write path would widen the write surface and pull the plan/apply
key-divergence guard and tiny-first tiering into map-keyed PUTs. The adopt path
resolves each key exactly once, inline, with no apply-time re-resolution, so a
map-sourced key cannot trip a divergence guard the adopt path never runs. Extending
`--key-map` to upload is recorded as future work.

## Safety and scope

Additive. The default path (no `--key-map`) is byte-identical to v0.3.170: same plan
rows, same gate, same reporting shape plus inert new counters. No archive migration,
no id rewrite, no hash change. Every written delta still passes the adopt leak guard
and `safe_object_storage_remote_key`; a bucket/host/URL-shaped map key is refused.
`check_public_privacy` and `check_release_readiness` stay green.
