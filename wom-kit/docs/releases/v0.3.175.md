# v0.3.175 - force-reupload for live multipart, and multipart-proven upload tier2

v0.3.175 adds two live-verification aids to the object-storage upload adapter (Stage 2). It is
additive: no migration, existing receipts and manifests are unaffected, and the DEFAULT
(flag-absent) path is byte-identical to v0.3.174.

## The two gaps

The basoon 5th operator letter (§3) surfaced two independent gaps that make a **live** R2
multipart demonstration impossible today.

**GAP A — no way to re-PUT an already-present object.** `object_storage_upload_run`'s F0-b
step re-HEADs the recorded `remote_key`; on present+match it emits `skipped_remote_same` and
CONTINUES. The only object > 5 MiB the client has is already uploaded, so it can never be
re-PUT to exercise a live provider multipart.

**GAP B — tier2 is a hard 5 GiB byte compare.** `object_storage_proven_tier` proved upload
tier2 ONLY by `bytes_uploaded >= OBJECT_STORAGE_MULTIPART_THRESHOLD_BYTES` (5 GiB). A forced
SMALL multipart (`part_count > 1`, already possible via `--multipart-part-size` /
`--allow-tiny-parts` since v0.3.172) has bytes < 5 GiB, so it was NOT recognized as the
multipart proof it actually is. A byte-external client with no > 5 GiB object could never
prove upload tier2.

## FIX A — approval-gated `--force-reupload`

`object-storage-upload --force-reupload` RE-PUTs an already-present, size/hash-matching object
so a client can exercise a LIVE provider PUT (e.g. a forced small multipart). It is a
**sequencing** change — which short-circuits are skipped — never an **integrity** change.

What it does:

- **Requires `--approve` AND `--reviewed-by`** — a deliberate provider PUT cost and a live
  overwrite. Enforced at BOTH the CLI and the service layer (a direct service/MCP call
  re-hits the same gates), and **inert under `--dry-run`** (refused with a clear message).
- **Refused for any non-sha-derived `--key-strategy`** (`force_reupload_requires_sha_derived_key`).
  The conflict-guard bypass is safe ONLY because both current strategies
  (`sha256_content_addressed`, `prefix`) embed the full 64-hex digest in the remote key
  (hard-enforced by the `clean not in key` raise in `object_storage_remote_key`). A future
  non-sha strategy would make force-reupload a real clobber vector, so the whole flag is
  refused there.
- **Bypasses exactly two short-circuits** for the targeted object: (1) the F0-b present+match
  `skipped_remote_same` skip, and (2) the resume-ledger terminal-success short-circuit (via
  the existing `force_upload` mechanism). The redundant HEAD-before is also suppressed — that
  is the existing `force_upload` behavior, not a new bypass.
- **Preserves every integrity invariant.** Control still falls THROUGH to the unconditional
  pre-PUT local re-verify `sha256(local) == object_id`; a corrupt local file returns
  `failed_upload` and is NEVER PUT. The HEAD-after GET-rehash verification runs exactly like a
  first PUT, the orphan `delete_object` on a HEAD-after mismatch (SA-5) still fires, and the
  cumulative `OBJECT_STORAGE_TOTAL_PUT_CEILING` still bounds the run.
- **Records `forced_reupload` in the execution receipt** — a bare top-level boolean, `True`
  only on a real forced re-PUT, `False` on every default run.
- **The forced receipt is exempt from the C7 manifest-linkage requirement.** A forced re-PUT
  lands a SECOND `uploaded` receipt at the SAME content-addressed `remote_key` of an object
  that already carries its lone `wom_uploaded` manifest location (which links the ORIGINAL
  receipt). The manifest keeps exactly one `wom_uploaded` location per `remote_key` (the append
  is idempotent-per-key), and a forced re-PUT is by design a live-verification action, not a
  new manifest transition. The Doctor / execution-receipt validator (C7) therefore EXEMPTS a
  receipt with `forced_reupload=true` from the "a success receipt must be linked by a
  `wom_uploaded` location" rule, so a legitimate forced receipt no longer produces a spurious
  `object_storage_upload_wom_location_missing` ERROR. Every other receipt invariant (schema,
  `operation`, `reviewed_by`, `remote_key` digest bind, `retry_summary` scalar-only) still
  applies to the forced receipt; only the manifest-linkage requirement is relaxed for it.

Operator use:

```
# tiny-first has proved upload tier 1 for this store; the object is already present.
object-storage-upload --only <one-sha> --approve --reviewed-by <actor> \
    --force-reupload --multipart-threshold <bytes> --multipart-part-size <bytes> \
    --allow-tiny-parts        # a LIVE forced small multipart proving tier2
```

## FIX B — multipart-proven upload tier2

`object_storage_proven_tier` now proves tier2 by a genuine 5 GiB large-object PUT (kept
verbatim as the first disjunct) **OR** a real multipart execution (`part_count > 1` on an
`uploaded` receipt). A forced small multipart is now the tier2 proof it actually is, and
small-object upload tier2 verification is incidentally opened.

The new disjunct is guarded to `result_status == "uploaded"`: `part_count > 1` is only ever
written from `_object_storage_multipart_put` on an `uploaded` receipt (skip statuses carry
`part_count 0`), so a fabricated `skipped_remote_same` receipt with `part_count > 1` can never
mint tier2. `part_count > 1` is unforgeable in-process — it counts real `put_part` calls over
real `handle.read()` chunks, verified by HEAD-after before it reaches a receipt.

Because the new disjunct sets the SAME `saw_multipart_proof` flag as the 5 GiB path, a forced
small multipart + `>= OBJECT_STORAGE_TIER3_BATCH_MIN` distinct successful objects now reaches
tier3 without any 5 GiB object. This is intentional and tested; no consumer treats
`proven == 3` as stronger than `proven == 2`, so no gate is loosened.

## Safety envelope

- **No clobber of wrong bytes.** To PUT, control passes the unconditional local re-verify
  `sha256(local) == digest`; the bytes PUT always match the digest embedded in the
  content-addressed remote key. Force-reupload is refused for any non-sha key, so the
  conflict-guard bypass is a *repair* (sha-verified-correct bytes at a content-addressed key),
  never a clobber vector.
- **No runaway billing from FIX B.** Even when a forced small multipart lifts `proven_tier` to
  2 and passes the tier gate, batch cost is bound by the HARD cumulative
  `OBJECT_STORAGE_TOTAL_PUT_CEILING = 64`, not by the tier gate and not by `--max-objects`
  (Optional / refuse-only). The tier gate is a precondition; the ceiling is the cost bound.
- **Adopt tier ladder unaffected.** `object_storage_adopt_requested_tier` /
  `object_storage_adopt_proven_tier` (v0.3.174, R1) are byte-identical and structurally
  independent of `part_count` / `bytes_uploaded`, so a FIX-B forced-multipart UPLOAD receipt
  (which carries no `adopt_verification`) can never lift adopt proven tier above 0.
- **Default path byte-identical.** With the flag absent, all new CLI/service gates are no-ops,
  the F0-b present+match skip fires exactly as before, and the receipt gains only the
  always-present `forced_reupload: false` boolean, which trips no validator or leak gate.

## Tests

Fifteen tests (13 new + 2 extended/kept) via the injected fake transport — no real socket:
multipart `part_count > 1` reaches tier2; forced multipart + distinct ids reaches tier3; a
forged skip `part_count` stays tier1; the 5 GiB path stays recognized; the R1 adopt/requested
tier functions stay byte-identical; `--force-reupload` re-PUTs a present+match object and
records `forced_reupload`; a forced small multipart records `part_count > 1` and proves tier2;
the default no-flag path still skips; a corrupt local file is refused before any PUT; a
HEAD-after mismatch deletes the orphan; refused without `--approve` / without `--reviewed-by`
/ for a non-sha key; the receipt passes the schema, the C7 validator, and the §6 privacy gate;
and the PUT ceiling still bounds a forced re-PUT.

## Compatibility

Additive; **no migration required**. Existing receipts and manifests are unaffected. The full
test suite, `check_public_privacy`, and `check_release_readiness` stay green.

## Upgrade

See `UPGRADE.md`. All changes are additive; no migration is required. The only new behavior is
the opt-in `--force-reupload` (approval + reviewer gated) and that a real multipart is now
recognized as an upload tier2 proof.
