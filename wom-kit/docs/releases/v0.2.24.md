# v0.2.24 Block Header Preview

Date: 2026-05-25

## Summary

v0.2.24 adds a read-only dry-run preview for deriving a block header from one existing draft or canonical zet.

The model is:

```text
block = zet + header
```

The zet remains the minimum human-supervised text information unit. The header is derived from refs, hashes, provenance, policy, receipts, source refs, and objet refs. ZET remains the sharing layer for delegate, attest, and anchor flows.

## Added

- CLI `archive block-header <archive-root> --path <zet-path> --dry-run --format json`.
- CLI `archive block-header <archive-root> --zettel-id <id> --dry-run --format json`.
- Deterministic `zet_body_sha256`, `header_sha256`, and `block_hash_preview`.
- Header preview fields for source refs, source intake metadata, assets, objet refs, edges, referenced zets, mint/promotion metadata, receipts, provenance, and visibility.
- Read-only MCP `block_header_check`.

## Hash Semantics

`header_sha256` is computed over the sanitized, shareable header projection. Private local paths, provider URLs, and secret-like references are redacted before hashing. Therefore it is a preview commitment to the shareable block view, not a raw private-frontmatter integrity hash.

`zet_body_sha256` is also block-preview specific. It is not interchangeable with draft approval replay hashes such as `expected_body_sha256`.

## Boundaries

This release does not:

- write files,
- mint,
- modify existing zets,
- read referenced objet/source file bodies,
- calculate referenced source hashes,
- follow provider URLs,
- call provider APIs,
- implement real ZET transport,
- implement token mechanics, WOM coin, NFT-like access, paid delegation, staking-like trust, payments, consensus, ledgers, relays, P2P, or blockchain behavior.

## Upgrade Notes

No private archive migration is required.

Existing zets remain valid. `block-header` is a preview tool only and returns `would_change: []`.
