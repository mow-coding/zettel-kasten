# v0.3.119 - Batch Mint/Retire Robustness

Date: 2026-06-20

## What Changed

This release adds batch-safe lifecycle commands for large WOM real-use archive
maintenance:

```powershell
archive mint-zet-batch <archive-root> --plan workbench/mint-zet-batch.plan.json --dry-run --format json
archive mint-zet-batch <archive-root> --plan workbench/mint-zet-batch.plan.json --approve --reviewed-by person:me --format json

archive retire-draft-batch <archive-root> --plan workbench/retire-draft-batch.plan.json --dry-run --format json
archive retire-draft-batch <archive-root> --plan workbench/retire-draft-batch.plan.json --approve --reviewed-by person:me --format json
```

Aliases are available for shorter operator runs:

- `bulk-mint`
- `bulk-mint-zet`
- `bulk-retire`
- `bulk-retire-draft`

The commands consume one reviewed JSON plan, process many items inside one WOM-kit process,
and keep the existing single-item safety gates underneath each item.

## Why It Matters

The preceding releases made individual mint operations much faster by using the
generated SQLite index and standard zettel-id path resolution. Real DB2.0
migration work then found a different robustness gap: operators still had to
compose shell loops for thousands of `mint-zet` or `retire-draft` calls.

On Windows/Git Bash, that style can fail from process pressure before the
archive logic itself is the bottleneck. It can also tempt an AI/runtime to use
`find | xargs` helper loops that miscount already processed items.

v0.3.119 moves the batch boundary into WOM-kit itself:

- one process per batch command,
- one reviewed plan file,
- one batch receipt,
- itemized `failed_items` for partial failures,
- `--skip-existing` for safe reruns after interruption,
- `--max-items` as an explicit operator guardrail,
- no per-item shell process spawning.

## Plan Shape

A minimal mint plan:

```json
{
  "schema": "wom-kit/mint-zet-batch/v0.1",
  "policy": {
    "policy_id": "policy:batch-mint-reviewed"
  },
  "items": [
    { "zettel_id": "zet_20260620_example_one" },
    { "path": "inbox/zet_20260620_example_two.md" }
  ]
}
```

A minimal retire plan:

```json
{
  "schema": "wom-kit/retire-draft-batch/v0.1",
  "policy": {
    "policy_id": "policy:batch-retire-reviewed"
  },
  "items": [
    { "zettel_id": "zet_20260620_example_one" },
    { "path": "inbox/zet_20260620_example_two.md" }
  ]
}
```

Each item must identify exactly one draft by `zettel_id` or archive-relative
`path`.

## Receipts

Approved mint batches write a batch receipt under:

```text
receipts/mint/batches/
```

Approved retired-draft batches write a batch receipt under:

```text
receipts/mint/retired-drafts/batches/
```

The existing per-item mint receipts, draft snapshots, and retired-draft receipts
are still preserved.

## Compatibility

No archive migration is required.

Existing single-item commands remain valid:

- `mint-zet`
- `mint-zettel`
- `retire-draft`
- `retire-minted-draft`

The batch commands are additive. They call the same underlying mint and retire
checks item by item, but they avoid starting a new shell process for every item.

## Safety Boundary

This release does not:

- bypass human `--reviewed-by` approval,
- mint drafts that fail the existing mint gates,
- retire drafts without canonical, receipt, snapshot, path, and SHA evidence,
- auto-generate batch plans,
- read unrelated source files,
- echo zettel body text,
- call external providers,
- add MCP write tools,
- add a background worker, daemon, queue, or scheduler,
- delete canonical zets, mint receipts, or draft snapshots,
- replace `index-health`, `doctor`, or `validate`.

If a batch stops partway through, fix the listed `failed_items` and rerun with
`--skip-existing`.

For a one-process status query before writing anything, run the same plan with
`--dry-run --skip-existing`. That returns which items would write, which items
are already covered by existing artifacts, and which items are blocked, without
using `find | xargs` counting loops.
