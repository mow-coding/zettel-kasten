# v0.3.211 - whole-response token observability

v0.3.211 shows the difference between catalog item cost and the full compact
service result, then lets a host reserve room for that response envelope. It is
additive and requires no archive migration.

## What Changed

- Added compact service-result, items, and response-envelope character, byte,
  and heuristic token measurements to every catalog page.
- Added CLI `--response-envelope-reserve-tokens` and MCP
  `response_envelope_reserve_tokens`.
- Added `coverage.effective_items_token_budget` and reserve adequacy/total
  budget observations.
- Added warnings and next actions when the measured compact result exceeds an
  opt-in requested total; exhaustive progress still returns at least one item.
- Extended the synthetic benchmark with response-budget and reserve checks.

## Scale Evidence

A 10,000-node seeded compact-reading pass used an 8,000-token total request,
2,500-token envelope reserve, and 5,500-token effective item budget. It
completed 264 pages in 58.1 seconds. The largest measured compact service
result was 6,879 tokens; no page exceeded the effective item budget, requested
response budget, or measured envelope reserve.

## Compatibility And Safety

Catalog output schema advances to v0.7. Continuation schema stays v0.3 because
page size, item budget, and reserve may change between strict pages without
changing coverage order. Existing calls that omit the reserve keep items-only
budget semantics.

Measurements use a four-characters-per-token heuristic. They exclude the
measurement block itself, CLI pretty whitespace, MCP/JSON-RPC framing, zet
bodies, and objet bytes. They are not provider-reported accounting or an exact
context-window guarantee.

No model/provider is called, no token state or host loop is persisted, and no
archive file is written.
