# v0.3.3 - Field Feedback Hardening

Release date: 2026-06-13

This is a compatible patch release for v0.3.2 field feedback. It tightens CLI
validation behavior, clarifies cleanup safety signals, makes list-valued zettel
facets queryable through `view-zets`, and strengthens objet-capture regression
coverage around real source-intake producer plans. The v0.3.1 frontmatter schema
does not change.

## Fixed

### Console and validation feedback

- CLI output no longer crashes on console encodings that cannot represent a
  character, such as emoji on a Korean Windows cp949 console. Unencodable
  characters are replaced instead.
- `archive doctor` warns with `zettel_frontmatter_unquoted_timestamp` when
  frontmatter contains an unquoted YAML timestamp. The warning includes the field
  path and a quoting hint.
- `archive validate --strict` is accepted for parity with doctor. Validate still
  fails on warnings unless `--allow-warnings` is used.
- `wom-kit/docs/validation-surface.md` now documents what doctor, validate,
  preflight, and staged cleanup checks each guarantee.

### Staged cleanup safety signal

- `archive staged-cleanup-check <archive-root> --staged <folder> --dry-run`
  remains report-only and never deletes files.
- The process exit status is now a safety signal:
  - exit `0` only when the check ran and `safe_to_cleanup` is `true`;
  - exit `1` when the check fails or the report says cleanup is unsafe.
- Automation should still read the JSON report for per-file details and should
  treat cleanup as approved only when `safe_to_cleanup` is `true`.

### Facet views

- `archive index` now indexes list-valued zettel facets as repeated scalar facet
  rows.
- `archive view-zets` can now match scalar facet filters against list-valued
  facets, such as `tags: [alpha, beta]`.
- List-valued filter inputs are blocked instead of guessed or broadened.
- Redacted zettels remain excluded from indexed facets and `view-zets` results.

### Objet capture evidence

- Objet-capture source-intake plan SHA binding is now regression-tested against a
  real `archive source-intake --dry-run --format json` producer plan.
- The test verifies dry-run acceptance, approve capture, and SHA mismatch
  blocking after the saved source-intake plan is tampered.
- The selection manifest field names remain unchanged:
  `source_intake_receipt_path` points to the saved plan JSON, and
  `source_intake_plan_sha256` binds the canonical JSON hash of that plan.

## Upgrade

No archive schema migration is required for v0.3.2 users.

If you rely on `view-zets` facet filters, rebuild the disposable search index
once so list-valued facets are expanded:

```text
archive index <archive-root>
```

If you automate staged-folder cleanup checks, update the automation to treat a
nonzero exit from an unsafe report as expected fail-closed behavior, and read the
JSON `safe_to_cleanup` field before any manual cleanup decision.

## Safety Boundary

This release does not touch live basoon archives, upload or sync providers,
delete staged originals or cleanup targets, create ZET transport, mutate trust,
redesign the v0.3.1 schema, or add MCP write tools. Cleanup remains a manual
human action outside WOM-kit.

See the [Upgrade Guide](https://github.com/mow-coding/zettel-kasten/blob/main/UPGRADE.md) and [Changelog](https://github.com/mow-coding/zettel-kasten/blob/main/CHANGELOG.md).
