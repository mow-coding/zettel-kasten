# v0.3.257 - Strict Revision Approval Snapshots

v0.3.257 makes revision and restore evidence fail closed. WOM-kit now validates
the complete zettel boundary, schema, identity, archive, and lifecycle before
the rejected bytes can become a SHA-256 value, equality result, plan digest, or
approved write candidate.

## What Changed

- Approval-bound YAML rejects duplicate mapping keys at every depth, requires
  string mapping keys, and normalizes only a bounded acyclic JSON value tree.
  Cyclic aliases, excessive depth/node count, YAML set/binary values, and
  non-finite numbers fail closed. Existing unquoted timestamp compatibility is
  preserved by normalizing dates to ISO strings. The tolerant parser used by
  import and capture workflows is unchanged.
- One bounded regular-file snapshot now binds the opened file generation to
  its path before returning any bytes. Invalid UTF-8, frontmatter delimiters,
  YAML, schema, id, archive id, or lifecycle status returns fixed issue codes
  with no bytes, frontmatter, or body.
- Revision plan and write paths use that validated snapshot for the current
  canonical zet and private proposal before computing file or semantic hashes.
  The same ordering is repeated after a write lock and during interrupted-write
  recovery.
- Revision receipt verification and history audit derive current-state evidence
  only from a validated canonical snapshot. An invalid current zet is reported
  as unavailable evidence, never as a hash comparison.
- Restore proposal materialization validates the retained before-snapshot before
  creating scratch output. Existing and newly copied restore proposals are
  validated before exact-state success is reported.
- Restore plan, write, recovery, and receipt verification validate current and
  proposed canonical generations before producing state hashes or equality
  claims.
- Revision and restore proposal/receipt path checks reject symbolic-link and
  Windows reparse-point components.

## Compatibility And Migration

- No archive schema or automatic zettel migration is required.
- Existing valid revision and restore commands keep their CLI arguments,
  receipts, plan schemas, and normal approval flow.
- A previously tolerated duplicate/non-JSON YAML value or
  schema/identity/status mismatch now blocks with content-free evidence fields.
  Repair the private proposal or source zettel deliberately, then create a
  fresh plan.
- Rejected plan results intentionally set zettel hashes, semantic hashes, state
  equality, and plan digests to null or false rather than fingerprinting
  untrusted bytes.

## Verify The Upgrade

```powershell
archive version <project-or-archive-root> --format json

archive zet-revision-plan <archive-root> `
  --zettel-id <zet-id> `
  --proposal .wom-scratch/revisions/<private-proposal>.md `
  --dry-run `
  --format json

archive zet-revision-receipt-audit <archive-root> `
  --dry-run `
  --format json
```

For a restore, regenerate or revalidate the private restore proposal and run a
fresh restore plan. Do not reuse a pre-upgrade digest if the new result differs.

## Privacy And Safety Boundary

- Rejected zettel bytes are not returned by the strict snapshot helper and are
  not used as trusted revision hashes, comparison states, plan digests, or
  candidates.
- Fixed blockers and public results do not echo titles, abstract text, body
  text, custom frontmatter values, reviewer ids, absolute paths, or provider
  locators.
- This release changes validation ordering only. It does not claim a new
  cross-platform transaction engine or alter the existing canonical replacement,
  receipt publication, lock, rollback, or recovery algorithms.

## Deliberate Follow-Up Boundary

An earlier combined implementation attempted to fix validation, mutation,
reconcile, abstract backfill, workpack, and handoff boundaries together.
Independent review stopped that release because the new in-place mutation
helper regressed Linux/Docker behavior and could tear a live file on hard exit.
That combined work remains an unreleased checkpoint.

The next mutation batch must independently design and verify complete staged
generations, durable pre-mutation journals, platform-correct atomic handoff,
retained directory/file ownership, and idempotent recovery. Abstract backfill,
reconcile, workpack, and approval handoff hardening remain separate maintenance
batches so none of those claims are implied by v0.3.257.

## Provenance And Verification

The ordering defect was identified in the independent Fable5 review following
v0.3.256. Verification covers rejected delimiter, UTF-8, duplicate-key,
non-string-key, cyclic/deep/non-JSON YAML, non-finite-number, schema, identity,
archive, status, and size cases; normal revision/restore planning,
approval, recovery, receipt verification, and audit compatibility; scratch
residue after rejected restore materialization; the complete WOM-kit suite;
packaged-resource synchronization; release-readiness checks; clean-wheel
installation; and public-URL reinstallation. Real archive use remains a
separate beta-client confirmation step.
