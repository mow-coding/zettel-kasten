# v0.3.239 - Approved Exact-Byte Canonical Restore

v0.3.239 closes the first evidence-backed canonical correction and recovery
loop for the Memento Problem. A complete old zet recovered from private backup
can now move from read-only planning through human approval to exact local
restoration and one durable chronological receipt event.

## What Changed

- Added CLI-only `zet-revision-restore-write` with dry-run and approval modes;
  aliases are `canonical-revision-restore-write` and `zet-restore-write`.
- Bound the immutable source receipt, actual current state, recovered file and
  semantic hashes, restore-plan and history-audit digests, event timestamp,
  policy, and change summary into one writer digest.
- Required a safe reviewer, full restore review, abstract/body-pair review,
  and changed-edge review when applicable.
- Reused the ordinary revision writer's per-canonical lock and atomically
  installed the reviewed restore proposal bytes without reserialization.
- Added a separate immutable restore-receipt schema and registered ordinary
  plus restore receipts and locks in one chronological audit chain.
- Added exact rollback for handled failures and exact-rerun recovery before
  canonical writing, after canonical writing, and after receipt creation.
- Registered reviewed restore abstract/body hashes as freshness evidence.
- Prevented later ordinary revisions from using an event time older than the
  newest receipt after a restore preserves historical `updated_at`.

## Upgrade And Compatibility

No archive migration is required. Restart the AI operator process. Existing
v0.3.235 ordinary revision receipts remain valid, and v0.3.238 chronological
audit semantics remain the history model.

Run `zet-revision-receipt-audit --dry-run` first. Recover complete old zet
bytes from a trusted private backup, run `zet-revision-restore-plan`, privately
review all three artifacts, preview `zet-revision-restore-write --dry-run`,
and only then approve the unchanged writer digest and event time.

## Safety And Claim Boundary

- The writer never reconstructs text from hashes.
- Exact restored bytes retain YAML order, BOM, newlines, body, frontmatter,
  and historical `updated_at`; the restore event time belongs to the receipt.
- Handled runtime failure restores exact pre-restore bytes and removes partial
  transaction files.
- A forced interruption leaves content-free evidence for an exact approved
  rerun; mismatched evidence remains blocked.
- Output and receipts store no title, abstract text, body text, or custom
  frontmatter value. Public output also echoes no private identity/path,
  reviewer, provider value, absolute path, or secret.
- No restore MCP writer, provider call, network call, object write, or database
  write was added.
- `applied` is local evidence, not proof of truth, backup completeness,
  external synchronization, legal clearance, or model understanding.
