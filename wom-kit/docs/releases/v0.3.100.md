# v0.3.100 - Notion Objet Manifest Locator Label

Date: 2026-06-17

## Added

- CLI `archive notion-objet-manifest-locator-label <archive-root>
  --object-id sha256:<hex> --locator-fingerprint sha256:<hex>
  --dry-run|--approve`.
- Alias `archive notion-objet-locator-label`.
- Public docs for the approved manifest locator label write:
  `wom-kit/docs/notion-objet-manifest-locator-label.md`.

## Why

`notion-objet-link-index` can find Notion locator fingerprints in zettel
bodies, but real object manifests may only say that an object came from a
Notion export. If the manifest does not preserve the locator fingerprint,
matching stays at 0 even when the human knows which object belongs to the
locator.

This release adds the narrow bridge: after human review, WOM-kit can write a
non-secret `provider_locator_sha256` label to one object manifest record.

## Behavior

- Requires either `--dry-run` or `--approve`.
- Requires `--reviewed-by` for approved writes.
- Requires exactly one matching `object_id` in `objects/manifests/files.jsonl`.
- Adds a locator hash under `provenance.provider_locator_sha256` when empty.
- Preserves an existing different hash and adds
  `provenance.provider_locator_sha256_values` for multiple reviewed locators.
- Writes a receipt under `receipts/objects/notion-locator-labels/`.
- Lets later `notion-objet-link-index` and `notion-objet-link-plan` match the
  object without storing or printing the raw provider locator.

## Safety

The new command is CLI-only. MCP exposes no matching write tool.

It does not read zettel bodies, rewrite zettel bodies, write `embed` edges,
call Notion, start OAuth, read real source exports, read object bytes, write
candidate records, upload, download, or create provider URLs.

Output avoids provider URLs, provider locator text, zettel body text, zettel
titles, frontmatter values, page titles, absolute local paths, account ids,
emails, tokens, and secret values.
