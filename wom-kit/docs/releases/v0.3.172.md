# v0.3.172 - live-multipart part-size override and strip-bom dry-run parity

v0.3.172 lands two verification-honesty fixes. Both are additive, need no migration,
and leave every default path byte-identical to v0.3.171.

## Item 1 — live-multipart part-size override (`--multipart-part-size` + `--allow-tiny-parts`)

### The honest problem

Before this release the multipart chunk loop read the module constant
`OBJECT_STORAGE_MULTIPART_PART_SIZE_BYTES` (64 MiB) directly, and the threshold
validator refused any `--multipart-threshold` below that same 64 MiB. So a
sub-64-MiB object could never split into more than one real part, and could never
even *enter* multipart. The v0.3.167 framing that a "small object can exercise
multipart" was not accurate for a *live* run.

To be precise about what was and was not already proven:

- The multipart **code path** (create → put_part → complete → abort-on-error) is
  ALREADY unit-tested via the injected fake transport — a 65 MiB fixture drives
  `part_count == 2` with no network.
- What was **unproven**, and what `--multipart-part-size --allow-tiny-parts` now lets
  an operator prove, is **live R2 multipart**: real per-part SigV4 signing and a
  `CompleteMultipartUpload` accepted by R2.

### What ships

- **`--multipart-part-size <BYTES>` on `object-storage-upload`.** Default 64 MiB
  (unchanged). An override is bounded to `[4096, 64 MiB]`. Below the 64 MiB default it
  requires `--allow-tiny-parts`. It changes ONLY `handle.read()` fragmentation.
- **`--allow-tiny-parts`.** The acknowledgment gate for a sub-64-MiB part size, so a
  tiny part size cannot be set by accident. Its blocker states the R2 reality plainly:
  real R2 rejects multipart parts smaller than 5 MiB except the last, so a tiny part is
  a fake-transport / local validation aid — a live tiny-part rejection is an upload
  **rejection** (failed status), never a silent integrity bypass.
- **Threshold-floor re-basing.** When a part size is supplied, the `--multipart-threshold`
  floor is compared against the *effective part size*, not the 64 MiB constant. This is
  load-bearing: to force multipart on a small object an operator lowers BOTH the
  threshold and the part size. Without the re-basing the threshold could never drop
  below 64 MiB and `is_multipart` would never be true — the feature would be dead on
  arrival.
- **Receipt honesty.** The execution receipt gains
  `effective_multipart_part_size_bytes` (additive; the schema has no
  `additionalProperties:false` and does not list it in `required`). `part_count` alone
  proves a split happened but not the granularity; recording the part size lets an
  auditor verify `ceil(size / part_size) == part_count` and prove the split was
  **forced**, not incidental to object size.

### Integrity is preserved, not weakened

Lowering the part size changes only how the file is fragmented for `handle.read()`.
The whole-object before-hash (`content_sha256`), the FULL-OBJECT sha256 handed to
`complete_multipart`, the HEAD-after full-object verify, the SA-5 delete-on-mismatch,
and the §6 leak gate are all byte-for-byte as shipped. There is no per-part-hash
acceptance path, no HEAD-after short-circuit, and no new integrity branch keyed on part
size. On any part-size / threshold / acknowledgment violation the run does not proceed
and `transport.put_calls` stays 0.

## Item 2 — strip-bom dry-run parity (remint AND retire reconcile)

### The precise premise

The reconcile classifier is **already BOM-insensitive**: canonical is read with
`utf-8-sig` (the BOM is consumed on read) and body comparison strips the BOM and folds
newlines. A pure-BOM file already classifies `format_drift`; a BOM + content edit
already classifies `content_change` — regardless of `--strip-bom`. The real gap was
**metadata**: `--approve --strip-bom` previewed the strip-intent fields
(`bom_stripped`, `bom_strip_note`), but the dry-run modeled no strip, so an operator
could not preview them before approving.

### The fix — surface strip-intent metadata only; classifier is a NO-OP

- `--strip-bom` on a **dry-run** now previews the same strip-intent metadata an
  `--approve` run records: BOM present → `bom_stripped=True`, "would strip one leading
  UTF-8 BOM"; no BOM → `bom_stripped=False`, "no leading BOM present; nothing stripped"
  (a documented no-op — never a previewed byte rewrite).
- The strip preview is a **classification no-op**. `drift_class` and
  `content_change_ack_required` are identical whether `--strip-bom` is passed or not,
  for every file. The Tier A/B/C classifier is untouched. A real `content_change` is
  never laundered to `format_drift`.
- **Both surfaces are fixed**, not just remint: `retire-draft-reconcile` had the
  identical dry-run/apply asymmetry, and `--strip-bom` is registered on both parsers.

## Safety and scope

Additive. No new flag is on by default; the default reconcile and upload paths are
byte-identical to v0.3.171. No archive migration, no id rewrite, no hash change. The
new receipt field is optional in the schema. `check_public_privacy` and
`check_release_readiness` stay green.
