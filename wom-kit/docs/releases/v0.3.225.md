# v0.3.225 - Targeted Full-Doctor Edge Receipt Index

v0.3.225 removes the reproduced all-receipt JSON bottleneck from historical
mint target SHA validation. It is additive and requires no archive migration.

## What Changed

- Doctor builds one safe filename-only edge-receipt index per run.
- A mismatched zet loads only receipts named for its source id and any legacy
  receipt paths referenced directly by its current edges.
- Loaded candidates must still match the receipt kind, source path, and valid
  creation time before they can explain SHA evolution.
- Per-source results are cached across mint and retired-draft checks.
- Progress exposes separate `edge-receipt-index` and
  `edge-receipt-source-load` counts without content-bearing values.
- A temporary synthetic benchmark reproduces the 8,583-receipt scale and
  verifies one Doctor index build, one source load, and one cache hit.

## Upgrade Guidance

No command changes are required. Run a complete inspection explicitly when it
is needed:

```text
archive runtime-context <archive-root> --full-doctor --progress --format json
```

During historical target SHA checks, expect the outer mint count to hand off
briefly to an `edge-receipt-index` total, followed by a usually much smaller
`edge-receipt-source-load` total.

## Privacy And Work Boundary

The optimization reads local receipt filenames and only the selected local
receipt JSON candidates. It calls no provider or credential store and writes no
archive state. Shared progress includes counts and fixed stage labels only; it
does not include receipt paths, zet ids, edges, titles, bodies, abstracts,
provider values, or secrets.
