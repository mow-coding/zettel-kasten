# v0.3.186 - adopt stop-after-plan and doctor receipt continuation progress

v0.3.186 responds to the basoon v0.3.185 revalidation.

The revalidation confirmed that the v0.3.185 nonmatching-provider adopt summary
works. It also found two remaining operator pain points:

- a short `--approve` run still writes partial manifest/receipt state while the
  operator is only trying to inspect the plan-time summary;
- after `target mint receipt link ok`, `doctor --progress` could stay quiet long
  enough that the next bottleneck was still unclear.

This release is additive. Defaults remain conservative.

## Adopt stop-after-plan

`object-storage-adopt-existing` now accepts `--stop-after-plan`.

Use it when you want the exact plan-time summaries from an approve-shaped
command, but no provider calls and no writes:

```bash
archive object-storage-adopt-existing <archive-root> \
  --provider-kind cloudflare-r2 \
  --store-ref <store-ref> \
  --access-key-id-ref env:WOM_R2_ACCESS_KEY_ID \
  --secret-access-key-ref env:WOM_R2_SECRET_ACCESS_KEY \
  --endpoint-host <account>.r2.cloudflarestorage.com \
  --bucket <bucket> \
  --key-map ./key-map.jsonl \
  --reviewed-by person:me \
  --approve --skip-existing-wom-uploaded --stop-after-plan --progress --format json
```

With `--stop-after-plan`, WOM stops after `adopt-plan` and returns
`execution_status: plan_only`. It does not read credential values, issue
provider HEADs, update the object manifest, or write object-storage execution
receipts. The final JSON/text result is written to stdout. Optional `--progress`
heartbeats are written to stderr.

## Same-store wom_uploaded diagnostics

The basoon run showed a possible confusion between a simple manifest count and a
strict resume-skip count. A row can share the same `store_ref` and
`availability: wom_uploaded`, yet still fail the stricter gate for this run
because its key hint, resolved `remote_key`, or digest binding differs.

`adopt_summary` now includes:

- `existing_matching_wom_uploaded_location_count`
- `same_store_wom_uploaded_location_count`
- `same_store_wom_uploaded_gating_location_count`
- `same_store_wom_uploaded_non_gating_location_count`
- `same_store_wom_uploaded_key_hint_mismatch_count`
- `same_store_wom_uploaded_remote_key_mismatch_count`
- `same_store_wom_uploaded_remote_key_unsafe_count`
- `same_store_wom_uploaded_digest_unbound_count`
- `planned_remote_head_count`
- `unresolved_remote_key_count`
- `plan_only`
- `plan_only_stop_stage`

When non-gating same-store `wom_uploaded` rows exist, progress emits:

```text
[object-storage-adopt] adopt-plan: resume same-store wom_uploaded diagnostic store_ref_wom_uploaded_locations=<n> gating_locations=<n> non_gating_locations=<n> key_hint_mismatch=<n> remote_key_mismatch=<n> remote_key_unsafe=<n> digest_unbound=<n>
```

The warning `same_store_wom_uploaded_non_gating_gap` explains that a raw
`store_ref` count can be higher than the matching resume skip candidates.

## Doctor mint-receipt continuation progress

`doctor --strict --progress` now emits detailed mint-receipt sub-steps for the
first three mint receipts, not just the first one. For detailed receipts it also
prints:

```text
completed receipt checks
```

This makes a stall after `target mint receipt link ok` easier to localize: the
operator can now tell whether WOM completed the first receipt and moved into the
next receipt, or whether it stalled before completing the receipt.

## Tests

Added focused regression coverage for:

- `--stop-after-plan` returning `execution_status: plan_only`;
- `--approve --stop-after-plan` reading no credential values, issuing no provider
  calls, and writing no manifest/receipt files;
- same-store raw `wom_uploaded` counts versus strict gating counts;
- the new doctor `completed receipt checks` progress marker.

## Compatibility

Additive; **no migration required**.
