# v0.3.221 - Archive-Wide Abstract Receipt And Lock Audit

v0.3.221 closes the locally provable abstract-revision health loop with one
bounded, content-free audit. It is additive and requires no archive migration.

## What Changed

- Added `zet-abstract-backfill-receipt-audit`
  (`abstract-backfill-receipt-audit`).
- Verifies every bounded applied receipt as currently applied or paired with one
  valid revert receipt/current reverted state.
- Blocks malformed/diverged applied evidence and orphan revert receipts.
- Classifies completed-operation lock residue as a warning and a lock without
  matching completion evidence as an unresolved-transaction blocker.
- Reads lock filenames but never lock content.
- Scans up to 5,000 receipts and 5,000 locks; returns at most 500 bounded problem
  rows and commits all outcomes to one `audit_digest`.
- Collapses healthy lifecycles into counts instead of repeated response rows.

## Privacy And Write Boundary

The command reads private receipt metadata and canonical bytes for hash checks,
but echoes no receipt path, zet id/path, body, abstract, reviewer, lock content,
or local absolute path. It calls no model/provider/secret store.

It writes and deletes nothing. It does not clean locks, edit immutable receipts,
repair canonical zets, recreate proposals, or decide semantic abstract quality.

## Honest Boundary

Green proves bounded local consistency only. It is not abstract truth/quality,
remote backup, external-editor exclusion, or forced-termination recovery. More
than 5,000 supported receipts or locks blocks complete coverage and needs a
future paged audit.
