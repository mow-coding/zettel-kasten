# v0.3.0 - Shared Update Attestation Review Write

Release date: 2026-06-03

v0.3.0 opens the first narrow receiver-side write boundary after the v0.2.x freeze.

## Added

- CLI `archive shared-update-attestation-review <archive-root> --record <path> --decision <attest|needs_more_review|reject> --reviewed-by <actor> --approve --format json`.
- Local service `record_shared_update_attestation_review`.
- Deterministic record path under `shared-updates/attestation-reviews/`.
- Deterministic receipt path under `receipts/shared-updates/`.
- Replay/overwrite refusal for this release.
- Rollback if receipt writing fails after the review record is created.
- Documentation: [Shared Update Attestation Review Write](https://github.com/mow-coding/zettel-kasten/blob/main/wom-kit/docs/shared-update-attestation-review-write.md).
- Work log: [Shared Update Attestation Review Write Work Log](https://github.com/mow-coding/zettel-kasten/blob/main/wom-kit/plans/work-log-2026-06-03-shared-update-attestation-review-write.md).

## Behavior

The command first reuses `zet_shared_update_record_review_preview`.

If the preview blocks, the write blocks.

If approval is present and the preview passes, the command writes exactly two files:

```text
shared-updates/attestation-reviews/<case-id>.json
receipts/shared-updates/<case-id>.shared-update-attestation-review.json
```

The output uses archive-relative paths.

## Safety Boundary

This release does not implement real ZET transport, key creation, key-sharing registry, radio-frequency access creation, mirroring delivery, neighbor feed update, automatic renewal, trust graph mutation, import, acceptance, anchor/apply, signatures, public proof anchoring, DID/wallet/key custody, provider sync, WordPress publishing, projection writes, projection receipts, queues, workers, payment, staking, consensus, blockchain, token, system token, model training, backpropagation, or full-auto behavior.

MCP exposes no write/apply/approve sibling tool for this boundary.

## Upgrade

The shared-update attestation/review command itself requires no provider,
transport, or shared-update record migration.

Archives authored from older `wom-kit/zettel-kasten-rules/v0.2-draft`
frontmatter guidance may still need a frontmatter compatibility preview before
strict v0.3 validation:

```text
archive migrate <archive-root> --target frontmatter-v0.3 --dry-run
```

See the [Upgrade Guide](https://github.com/mow-coding/zettel-kasten/blob/main/UPGRADE.md) and [Changelog](https://github.com/mow-coding/zettel-kasten/blob/main/CHANGELOG.md).
