# v0.3.235 - Canonical zet Revision Write

v0.3.235 completes WOM's first ordinary canonical correction path. A reviewed
private full-zet proposal can now move from the v0.3.234 read-only plan through
an exact write preview to one approval-gated atomic replacement and immutable
revision receipt.

## What Changed

- Added CLI `zet-revision-write` with aliases `revise-zet-write` and
  `canonical-revision-write`; MCP remains read-only for this workflow.
- Required canonical SHA-256, proposal SHA-256, proposal semantic SHA-256, and
  the original revision `plan_digest` from `zet-revision-plan`.
- Added a second dry-run digest that binds normalized `revision_at`, the exact
  writer-produced candidate, its abstract/body pair, and fixed change
  categories before approval.
- Required a safe reviewer, complete-revision review affirmation, and explicit
  abstract/body-pair review; edge changes require a separate affirmation.
- Revalidated both files and both plan digests after acquiring one private
  per-canonical-zet write-ahead lock and immediately before mutation; distinct
  plans for the same zet serialize through the same lock.
- Added atomic canonical replacement, immediate byte verification, runtime
  rollback, partial-receipt cleanup, and interruption-safe receipt completion.
- Added immutable digest-named receipts under
  `receipts/revisions/canonical/` with text-free before/after hashes and human
  review evidence.
- Registered approved revision receipts as abstract-freshness evidence.

## Upgrade And Compatibility

No archive migration is required. Existing canonical zets and receipts remain
valid. Run `zet-revision-plan`, then `zet-revision-write --dry-run`, and only
then reuse the returned timestamp and write-plan digest with `--approve`.

## Safety And Claim Boundary

- Dry-run writes nothing.
- Approval writes one canonical zet, one immutable receipt, and one temporary
  private lock that is removed after success.
- Ordinary runtime failure restores the exact original canonical bytes and
  removes partial transaction files.
- A process interruption after replacement can finish the missing receipt by
  rerunning the exact same approved command; mismatched locks remain blocked
  for human inspection.
- CLI output and receipt evidence store no title, abstract text, body text, or
  custom frontmatter value. CLI also echoes no actual zet id/path, proposal
  filename, reviewer id, provider URL, absolute path, or secret.
- `applied` is evidence of reviewed local application, not factual truth,
  completeness, external synchronization, backup, legal clearance, or model
  understanding.
