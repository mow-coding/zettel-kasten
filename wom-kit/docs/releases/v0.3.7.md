# v0.3.7 - Intake Receipt Gates And Human Artifact Planning

Release date: 2026-06-13

v0.3.7 is a compatible fast-follow to v0.3.6. It carries reviewed
project-intake receipt context further through source-intake, draft/mint
metadata, and local objet capture, starts read-only human artifact store
planning for user-selected apps, and clarifies large derived-text batch review
output.

## Added

- CLI `archive objet-capture <archive-root> --project-intake-receipt <receipt>`
  context validation for dry-run and approved capture.
- Selection-manifest support for `project_intake_receipt_path`; when the CLI
  flag and manifest value are both present, they must match.
- Fake-archive roundtrip regression covering:

```text
project-intake-plan
-> project-intake-decisions
-> project-intake-status
-> source-intake --project-intake-receipt
-> saved source-intake plan
-> create-draft --source-intake-plan
-> mint-zet dry-run metadata preview
-> objet-capture --project-intake-receipt
```

- CLI `archive human-artifact-store --dry-run` for read-only planning of
  WordPress, Joplin, Notion, Obsidian, Evernote, generic Markdown, and generic
  workspace surfaces.
- MCP `human_artifact_store_plan` for the same read-only human artifact store
  contract preview.
- Per-item `item_status` in derived-text batch output so JSONL review can
  distinguish `ready`, `skipped`, `blocked`, and `written` rows at a glance.
- `project-intake-status` `next_review_prompts` for missing checklist ids so
  AI-assisted intake sessions can ask the next human-review questions without
  inventing or echoing answer values.

## Changed

- `objet-capture` now validates project-intake receipt context before staged
  bytes are read when a project-intake receipt is supplied.
- `create-draft --source-intake-plan` now preserves validated
  `project_intake_context` in draft `source_intake` metadata and later mint
  preview/receipts without copying decision answer values.
- `project-intake-status` now shows missing-question prompts in text output and
  JSON while keeping recorded decision values hidden.
- Derived-text batch docs now explain `items[]`, `manifest_line`, `item_id`,
  `planned_action`, `blockers`, and `warnings`.
- Project-intake docs and README now show the intended manual migration spine
  from project planning through source-intake, objet capture, derived-text
  registration, drafting, minting, and staged cleanup verification.
- `.gitignore` repair docs now explain that local collaboration/runtime
  patterns such as `/collab/` and `/.mow-harness/` are defensive local-only
  guardrails for mailbox, prompt, status, and secret-bearing workspace files.

## Boundaries

- The project-intake receipt is session evidence only. It does not authorize
  automatic import, capture, derived-text registration, draft creation, minting,
  provider calls, uploads, or cleanup.
- `human-artifact-store` and `human_artifact_store_plan` are read-only planning
  surfaces. They do not start OAuth, call providers, create notes, update notes,
  delete notes, publish WordPress posts, upload files, mint zets, clean staging
  folders, or run ZET transport.
- `derive-text capture --from-manifest` still registers only already extracted
  UTF-8 text. It does not run OCR, ASR, parsers, LLM vision, provider APIs,
  drafting, or minting.
- No archive migration is required for v0.3.6 users.

## Upgrade

No migration is required.

To review a human-guided project migration path, start with:

```text
archive project-intake-plan <archive-root> --staged-folder <folder> --dry-run
```

Then use the recorded decisions receipt as context for one-item-at-a-time
source intake and explicit local capture:

```text
archive source-intake <archive-root> --dry-run --project-intake-receipt <receipt> ...
archive objet-capture <archive-root> --selection <manifest.json> --dry-run --project-intake-receipt <receipt>
```

To plan a user-facing note/workspace/publication surface without connecting it:

```text
archive human-artifact-store <archive-root> --dry-run --surface-kind joplin
```
