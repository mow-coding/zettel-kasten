# v0.3.249 - Before-Snapshot Restore Proposal Bridge

v0.3.249 connects the exact local prior-byte artifact introduced in v0.3.248
to WOM's existing human-reviewed canonical restore workflow.

## What Changed

- New CLI-only `zet-revision-restore-proposal-from-snapshot` and alias
  `zet-restore-proposal-from-snapshot`.
- It requires a healthy canonical revision audit, an immutable v0.2 ordinary
  revision receipt, the exact expected receipt SHA-256, and a verified local
  before-snapshot plus matching object-manifest record.
- No-write preview derives a SHA-named destination under private restore
  scratch and returns a deterministic `plan_digest`.
- Approval must bind that unchanged digest and creates only one exact private
  proposal copy.
- The copy has a different storage identity from the immutable snapshot. It is
  never a hard link to historical object bytes.
- Existing exact independent copies are idempotent. Any other destination
  collision blocks and is not overwritten.
- `ai-artifact-inventory` classifies the private copy as
  `zet_revision_restore_proposal` instead of an ordinary AI working note.
- The returned proposal proceeds through the unchanged
  `zet-revision-restore-plan`, private human review, and
  `zet-revision-restore-write` approval sequence.

## Why It Matters

v0.3.248 made old canonical bytes durable, but operators still had to copy them
by hand into the restore scratch area. That manual gap was easy to misunderstand
and made an otherwise complete artifact trail awkward to use. The new bridge
creates a reviewable working copy directly from verified local evidence while
preserving the distinction between historical artifact, mutable proposal, and
approved canonical state.

## Safety Boundary

- Materialization is not review or restore approval.
- Canonical zets, immutable receipts, object bytes, and manifests are not
  modified.
- The private destination is content-addressed and never overwritten.
- The proposal is independently copied so editing it cannot alter the snapshot.
- Legacy v0.1 revision receipts remain valid history but cannot use this bridge;
  they still need separately recovered private bytes.
- No model, provider, remote object store, database, credential store, or
  network is called.
- No remote backup, factual truth, usefulness, or model understanding is
  claimed.

## Verification

Regression coverage includes no-write preview, stale-plan rejection, exact
independent copy creation, idempotent replay, handoff into the existing restore
planner, collision non-overwrite, hard-link rejection, v0.1 refusal, content
redaction, parser capability discovery, and the existing revision/restore
suite. Release gates also cover clean-wheel installation, package resources,
public privacy, release readiness, and the full WOM-kit test suite.
