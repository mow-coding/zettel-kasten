# v0.3.138 - Notion Recover File-Ref Credential Handoff

v0.3.138 reduces the real-use credential-entry pain in `archive
notion-recover` without changing the actor boundary.

## What Changed

- `archive notion-recover --credential-ref file:<local-token-file>` now works
  as a CLI-only fallback.
- The local wrapper extracts a Notion integration token from the selected text
  file and passes it through the same approval-gated Notion ancestor fetch
  chain.
- The wrapper restores any prior transient process value after the run.
- Dry-run output, beginner setup guidance, the capability matrix, README
  summaries, and this release note now explain the difference between:
  - implemented file-ref fallback,
  - hidden local prompt fallback,
  - planned vault/keyring one-click handoff.

## Why It Matters

The v0.3.136 one-command wrapper removed most of the developer command chain,
but a non-developer could still get stuck at token entry.

This release lets a user who already has a local token text file point to that
file instead of pasting a token into a hidden terminal prompt or building an
environment-variable command.

## Safety Boundary

The AI still does not receive the secret.

The `file:<path>` value is consumed only by the local CLI wrapper. Command
output and receipts do not echo:

- token file paths,
- token file names,
- transient env refs,
- token values,
- approval receipt paths,
- provider URLs,
- account ids,
- emails,
- secret values.

Vault/keyring one-click handoff remains the correct long-term credential-broker
direction, but live vault or OS keyring reads are not implemented in this
wrapper yet. `keyring:` and `secret:` refs fail closed today instead of
pretending a vault was opened.

## Upgrade Notes

No archive migration is required.

Existing `env:` based automation continues to work. The new `file:<path>`
fallback is intended as a local bridge for humans who already have a token file;
the recommended long-term storage target remains a password vault or OS keyring.
