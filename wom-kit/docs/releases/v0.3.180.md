# v0.3.180 - adopt-plan and doctor receipt cache

v0.3.180 hardens large-archive performance after the basoon v0.3.179 revalidation isolated two
stage-level bottlenecks: `object-storage-adopt-existing` was slow during `adopt-plan` before any
provider HEAD checks, and `doctor --strict --progress` was slow during `mint-receipts`.

This release is additive. Command shapes, result JSON, receipt formats, and progress output are
unchanged.

## `object-storage-adopt-existing` plan index

Large key-map adopts now build one per-run manifest index keyed by lowercase `object_id` before
the plan loop. The loop then does O(1) lookups instead of repeatedly calling
`find_manifest_record`, whose historical single-object API scans `objects/manifests/files.jsonl`.

The index preserves the old first-record-wins behavior of `find_manifest_record`.

This targets the reported 19k-object pattern where progress showed:

```text
[object-storage-adopt] adopt-plan: 600/19059 resolved object ...
```

and the ETA was dominated by plan resolution before verified remote HEAD checks began.

## Doctor per-run caches

`archive doctor` now keeps per-run caches for:

- file SHA-256 values;
- zettel frontmatter parsed from canonical/draft files;
- zettel BOM evidence.

Receipt checks reuse that evidence within one doctor run. This reduces repeated disk work in
`mint-receipts` and related receipt stages while preserving the same diagnostics and refusal
semantics.

## Tests

Added focused regression coverage that:

- `object-storage-adopt-existing` planning does not fall back to per-object
  `find_manifest_record` scans;
- doctor hashes the same mint receipt target file only once per run;
- doctor reuses cached zettel frontmatter rather than re-reading the same file.

## Compatibility

Additive; **no migration required**. Existing progress output remains safe stage/count output
only, and no object ids, remote keys, provider URLs, credential refs, tokens, or secrets are
added.
