# v0.3.217 - SHA-Bound Catalog Artifact Lifecycle

v0.3.217 closes the private scratch lifecycle introduced by the one-process
catalog pass. It is additive and requires no archive migration.

## What Changed

- `zet-catalog-pass` now reports `output.sha256` for the exact complete JSONL.
- Added `zet-catalog-pass-read` (`catalog-pass-read`) to stream and validate the
  complete artifact before optionally returning one selected page.
- Private page output requires the expected SHA-256 and is bounded to one page.
- Validation checks header/page/footer schemas, contiguous coverage, snapshot
  consistency, final revalidation, declared item projection, allowed fields,
  body-exclusion guards, and optional artifact identity.
- Added `zet-catalog-pass-cleanup` (`catalog-pass-cleanup`) with dry-run preview,
  required SHA-256 binding, explicit `--approve`, and required `--reviewed-by`.
- Added content-free byte progress and heartbeat for artifact reads.

## Honest Boundary

The JSONL can contain private ids, titles, abstracts, facets, ties, and edges.
It contains no zet bodies under this generated and validated contract. It must
not be committed or loaded whole into one model response.

Read validation writes nothing. Cleanup deletes only the selected complete
private scratch file and writes no receipt. It never deletes hidden partials,
archive records, objets, manifests, or provider state. SHA-256 is an identity
check, not a signature, model-comprehension proof, or abstract-completeness
claim.

## Verification

- Unsupported private result-field injection with a matching recomputed hash.
- Missing page, missing page hash pin, wrong cleanup hash, and conflicting mode.
- Cleanup preview preserves the file; approved matching cleanup deletes only
