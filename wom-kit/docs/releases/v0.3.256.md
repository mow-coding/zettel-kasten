# v0.3.256 - Fail-Closed Zettel Integrity And Honest Index Health

v0.3.256 prevents an invalid existing-archive zet from being mistaken for
ordinary readable body text and makes generated-index health count every safely
enumerated physical zet path.

## What Changed

- Existing archive zettels now pass one strict content boundary before their
  metadata or body is trusted. The boundary requires supported frontmatter
  delimiters, a YAML object, and lifecycle status `draft`, `canonical`,
  `archived`, or `redacted`.
- Invalid delimiter/YAML/object/status, UTF-8, and file-access states return
  fixed content-free issues. Direct read and adjacent core analysis/write gates
  block before returning content or changing archive state.
- Valid redacted zets keep only the small existence envelope needed by existing
  surfaces. Even that envelope now admits only validated scalar id/kind/time
  values. Their title, body, facets, visibility, links, hashes, counts, and
  derived analysis remain hidden.
- Search, views, facet reports, and related-zets allowlist readable lifecycle
  statuses. A legacy generated-index row with null or unknown status is hidden
  even before the next rebuild, and a dangling edge target with no zettel row
  is neither returned nor traversed.
- `archive index` installs a path/stat-only `unreadable` quarantine row for an
  invalid zet and removes its queryable logical content, edges, and facets. The
  rebuild finishes nonzero as `completed_with_quarantined_zettels` with
  `index_rebuilt: true` and `index_complete: false`.
- Generated-index metadata v0.2 persists `index_complete` and the quarantine
  count in the same transaction. An incomplete rebuild cannot be used as the
  mint/promotion duplicate oracle or by facet-scoped validation; repair plus a
  complete rebuild is required first. Live duplicate fallback also treats an
  unreadable canonical candidate as a blocker instead of skipping it.
- A metadata-complete index is not trusted on metadata alone. Before indexed
  duplicate content or facet scope is used, WOM-kit enumerates every physical
  zet and compares its stored file-size/mtime tuple. A changed, added, or
  deleted path makes the index stale and routes mint to the safe live fallback,
  while an unstatable, unenumerable, symlink, junction, or reparse uncertainty
  blocks approval. Scoped validation blocks for rebuild in either case.
- `index-health` keeps unreadable physical paths in the live count, reports
  separate readable-metadata and fixed issue counts/samples, and recommends
  source repair before rebuild. An invalid new file can no longer disappear
  from both sides of a false `current` comparison.

## Who Should Upgrade

Upgrade if an archive uses `read-zettel`, search, views, related-zets, catalog
or link-analysis surfaces, `archive index`, or `index-health`. The release is
especially important when an archive may contain a partially written,
mis-delimited, invalid-YAML, unknown-status, unreadable, or redacted zet.

## Compatibility And Migration

- No archive schema or automatic zet migration is required. The disposable
  generated-index metadata format advances to v0.2.
- Existing valid draft, canonical, archived, and redacted zets remain
  compatible.
- Strict inspection may reveal previously tolerated malformed or status-less
  files. Repair those source files deliberately; rebuilding the index cannot
  repair a source zet.
- A v0.3.255 generated index has no v0.2 completeness evidence and is not used
  for the current metadata fast path. Run one complete `archive index` rebuild
  after source health is clean.
- A completed index command with exit code 1 can now mean a safe but incomplete
  generated index was installed. Automation must inspect `ok`, `state`,
  `index_rebuilt`, `index_complete`, and the process exit code together.
- The generated index remains disposable. `index-health` remains explicit and
  never triggers a rebuild automatically.

## Verify The Upgrade

Run health first with a new private output filename:

```powershell
archive version <project-or-archive-root> --format json

archive index-health <archive-root> `
  --dry-run `
  --progress `
  --output .wom-scratch/diagnostics/index-health-v03256-before.json `
  --format json
```

If health reports `live_zettel_frontmatter_unreadable_or_invalid`, repair the
bounded archive-relative source paths before rebuilding. Then, only if health
still reports a missing or stale generated index, run:

```powershell
archive index <archive-root> `
  --progress `
  --output .wom-scratch/diagnostics/index-v03256-rebuild.json `
  --format json

archive index-health <archive-root> `
  --dry-run `
  --progress `
  --output .wom-scratch/diagnostics/index-health-v03256-after.json `
  --format json
```

A complete final result requires exit code 0, `index_state: current`, matching
live/indexed path counts, and no live frontmatter inspection issues. A rebuild
result with `completed_with_quarantined_zettels` is useful safety evidence, but
it is not a complete index and still requires source repair plus a new health
check.

## Privacy And Safety Boundary

- Fixed issue results include bounded archive-relative path/code samples, not
  titles, bodies, frontmatter values, parser excerpts, absolute paths, provider
  locators, or secrets.
- A malformed delimiter can require reading bytes past the intended
  frontmatter boundary to prove it invalid. The privacy guard reports that
  honestly; the bytes are never echoed.
- Quarantine sanitizes logical WOM query/API results. It does not claim
  forensic secure deletion from SQLite free pages, WAL files, filesystem
  snapshots, backups, storage media, or the source zet itself.
- Private `.wom-scratch/diagnostics` results remain local scratch, not receipts
  or canonical WOM records.

## Deliberate Follow-Up Boundary

This release covers the core fail-closed read, index, query, health, and
adjacent directly audited surfaces. It does not claim every approval-heavy
historical workflow already validates lifecycle and identity before hashes or
derived comparisons.

- Revision/restore, retire-reconcile, abstract-backfill, and target-workpack
  validation ordering are assigned to v0.3.257.
- Bounded-memory default S3-compatible upload/download transport is assigned to
  v0.3.258.

Those follow-ups are separate releases so each change can receive focused
regressions, independent review, and a clear operational result.

## Provenance And Verification

The defects were found in an independent post-v0.3.255 Fable5 audit and then
reproduced against the current source. Release verification requires malformed
redaction canaries across direct and indexed surfaces, unknown-status legacy DB
rows, unreadable physical-path health accounting, normal lifecycle
compatibility regressions, the complete WOM-kit suite, package-resource sync,
release-readiness checks, and clean-wheel installation. Real archive use remains
a separate beta-client confirmation step.
