# v0.3.219 - Approval-Gated Transactional Abstract Revision

v0.3.219 adds the reviewed write half of the missing-abstract workflow. It is
additive and requires no archive migration.

## What Changed

- Added `zet-abstract-backfill-write` (`abstract-backfill-write`).
- Requires exactly one of `--dry-run` and `--approve` plus the exact reviewed
  proposal SHA-256.
- A new approval additionally requires a safe `--reviewed-by` value and
  `--affirm-abstracts-reviewed`.
- Reruns the planner, rereads the proposal, and revalidates every canonical
  exact-byte hash before mutation.
- Adds only `frontmatter.abstract`, preserving BOM/newline/body bytes and all
  other parsed frontmatter semantics without changing `updated_at`.
- Restores every attempted canonical file byte-for-byte when an in-process item
  or receipt operation fails.
- Writes one deterministic private revision receipt containing target and hash
  evidence but no body or abstract text.
- Validates a matching receipt/current state as no-write `already_applied`.
- Bounds rollback snapshots to 16 MiB per canonical file and 256 MiB per batch.

## Privacy And Authority

Public output returns only safe status, row indexes, counts, modes, hashes,
blocker codes, and a digest-only receipt path. It does not echo reviewer, zet
ids/paths, titles, bodies, abstracts, proposal filename, or absolute paths.

The command calls no model/provider and never generates an abstract. Human
review is an explicit authority input, not something an AI may infer from a
green plan.

## Honest Boundary

The runtime transaction has a private short-lived lock and exception rollback,
but it does not lock external editors or write a crash-recovery journal. Forced
process termination, power loss, or machine failure can bypass in-process
cleanup. A leftover lock requires inspection rather than blind deletion.
