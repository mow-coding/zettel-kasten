# v0.3.181 - staged-cleanup-check progress

v0.3.181 addresses the operator-visible "silent wait" gap in
`archive staged-cleanup-check`. The command could be working through a large staged folder or a
large source/store hash without emitting any intermediate signal.

This release is additive. Default output, result JSON, exit semantics, receipt formats, and the
report-only cleanup contract are unchanged.

## `staged-cleanup-check --progress`

`archive staged-cleanup-check <root> --staged <folder> --dry-run --progress` now streams progress
to stderr while it:

- loads canonical object records;
- scans zettel references;
- walks staged entries;
- verifies staged file preservation;
- hashes large staged files and their stored copies.

Progress lines are content-free: stage names, counts, and large-file byte totals only. They do not
add staged file names, object ids, local absolute paths, provider URLs, tokens, or secret values.

## Large-file hash liveness

`sha256_path` now accepts an optional progress callback. Existing callers keep the same digest
behavior because the callback is opt-in. Staged cleanup uses the callback only to report large-file
hash progress, so a single very large file no longer looks completely idle from the outside.

## Tests

Added focused regression coverage that:

- staged cleanup service progress emits stages without file names or object ids;
- CLI `--progress` emits staged-cleanup progress lines;
- hash progress callbacks preserve the exact SHA-256 digest result.

## Compatibility

Additive; **no migration required**. Omit `--progress` for the previous output shape. Cleanup is
still manual and human-approved; this command still never deletes files.
