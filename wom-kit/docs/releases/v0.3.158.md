# v0.3.158 - Objet-Capture Enablement for Real Archives

v0.3.158 lets a real (non-sandbox) archive run local objet capture after
explicit, receipted, revocable owner enablement. Motivating case: a client
archive root whose name matches the external live-store protection pattern
(`zettel-kasten-*`) was double-blocked — the never-touch name check fired
before the sandbox check, and no graduation path existed.

## What Changed

- Added `archive objet-capture-enable <archive-root>` (alias
  `archive capture-enable`):
  - `--dry-run` is a read-only eligibility report with a `state` field
    (`enabled` > `revoked` > `invalid_record` > `sandbox_marked` >
    `not_an_archive` > `not_enabled`), an orthogonal
    `never_touch_name_match` boolean, `planned_writes`, and a `reason` string
    for invalid records; it writes nothing. `--dry-run` is also valid with
    `--revoke`.
  - `--approve --reviewed-by <actor>` evaluates every blocker BEFORE any
    write, then writes the receipt first and the singleton record
    `ops/capture-enablement.yml` second. Pattern-matched root names require
    `--acknowledge-never-touch-name` (recorded as
    `never_touch_acknowledged: true` in record AND receipt); re-approving over
    a revoked record requires `--reenable`.
  - `--revoke --approve --reviewed-by <actor>` sets `enabled: false` plus
    `revoked_by`/`revoked_at` and writes a revoke receipt. Revoking never
    requires `--reenable` or the acknowledgment flag.
  - The command is CLI-only and NOT exposed via MCP.
- Reworked the objet-capture gate order: a valid enablement record allows
  capture BEFORE the never-touch name check, so an owner-enabled root whose
  name matches the pattern can capture. Validity requires exact `schema`,
  `scope: "objet-capture"`, and owner `statement` matches, `archive_id`
  equality with `archive.yml`, `enabled is True` by boolean identity (quoted
  truthy strings and integers fail closed; unquoted YAML 1.1 `yes`/`on` parse
  to the boolean `true` and DO enable), and
  `never_touch_acknowledged: true` on pattern-matched roots. Any read or
  validation exception fails closed to a clean refusal, never a traceback.
- Made the per-item never-touch checks enablement-aware: for a validly-enabled
  root, the source, destination, and selection call sites evaluate the name
  pattern on the path components RELATIVE to the enabled root only, so the
  root's own name no longer re-blocks every item while an item whose OWN
  relative component matches (for example `staging/incoming/evil-objets/…`)
  still blocks with `resolved_path_never_touch`. Non-enabled roots keep the
  absolute semantics unchanged; `target_looks_external_live_never_touch`
  itself is unmodified in both copies (runtime and hygiene checker).
- Added the additive refusal field
  `enablement_state: "absent"|"invalid"|"revoked"|"disabled"` to the
  `sandbox_marker_required` and `external_live_never_touch` refusals.
- Updated both objet-capture refusal hints to name
  `archive objet-capture-enable <archive-root> --dry-run` (read-only) as the
  owner inspection route; the `sandbox_marker_required` hint no longer calls
  real-archive enablement "a separate planned flow".
- Added doctor diagnostics: INFO `capture_enablement_enabled` and
  `capture_enablement_revoked`; WARN `capture_enablement_record_invalid`
  (with the reason) and `capture_enablement_receipts_missing` (valid record,
  zero receipts). The WARNs fail strict validation and surface via MCP
  `archive_doctor` — intended.
- Swept stale guidance: source-intake next safe actions, the objet-capture
  argparse help, the capability matrix capture row, the project-intake session
  table, and the artifact-hygiene "Current Gaps" list now name the enablement
  flow.
- New public doc: `wom-kit/docs/capture-enablement.md` (consent model, record
  and receipt schemas, revocation lifecycle, Safety Boundary).

## Safety Boundary

Stated exactly, without overclaiming (full text in
`wom-kit/docs/capture-enablement.md`):

- The record is STRICTLY STRONGER than `.wom-sandbox`: the marker cannot
  override the never-touch name pattern, the record can. Graduation removes a
  previously-unbypassable name-based protection for exactly the guarded path
  class.
- Anyone with archive write access can forge enablement (a minimal
  `archive.yml` plus a matching record validates). The archive-id binding
  prevents cross-archive COPYING only when ids differ; clones/backups/forks
  sharing an id inherit enablement.
- The receipt proves the command ran with the given `--reviewed-by` string,
  not that a human consented.
- Revocation is advisory and forward-only: captured bytes, manifest records,
  and capture receipts remain; the record is last-writer-wins; `--reenable`
  keeps an operator from silently undoing a human revocation but is not an
  enforcement boundary.
- The gate does not require receipts: receipts are evidence, enforcement is
  the record. Real improvements over the marker, stated as such:
  approval-gated command, named reviewer, receipt trail, id binding, explicit
  never-touch acknowledgment, revocation signal, doctor visibility.

Refusal outputs stay blocker-only: `enablement_state` is a root-level
classification and no item provenance, filenames, or receipt paths are echoed.
When `ops/capture-enablement.yml` is absent the gate adds a single stat and
reads nothing; when present it reads at most the two control files
(`ops/capture-enablement.yml` and `archive.yml`).

## Upgrade Notes

No archive migration is required. No JSON fields are renamed. JSON consumers
of `objet-capture` refusals see one ADDED field, `enablement_state`; the
`blocked_by` ids `sandbox_marker_required` and `external_live_never_touch` are
unchanged. The hint TEXT changed for both objet-capture refusal hints — hints
are static strings, not a parsing contract; downstream copies of the
`"separate planned flow"` substring assertion need the same update. Per-item
never-touch semantics change for validly-enabled roots only; non-enabled roots
(including all sandbox-marked archives without an enablement record) behave
exactly as before.
