# v0.3.174 - adopt tier decoupled from the upload 5 GiB proof

v0.3.174 fixes a design error in the verified-adopt tiered gate. It is additive: no
migration, existing receipts and manifests are unaffected, and the object-storage-upload
tier ladder is left byte-identical.

## The deadlock

`object-storage-adopt-existing --approve` runs a live HEAD per object. To bound the first
live batch (a naive first-live run of 19,054 objects would issue 19,054 uncosted HEADs at
once), v0.3.166's F5 made a batch adopt honour the SAME SA-6 tiered gate as
`object-storage-upload`: a batch (requested upload tier 3) required a store proven to upload
tier >= 2, and upload tier 2 is a genuine **5 GiB multipart / large-object PUT** proof.

That coupling deadlocked the exact case adopt exists to serve. The basoon 158 GB handover
(its 5th operator letter, §2) is a batch of 19,054 objects **already present** at the
client's own key layout. There is no large object to PUT and prove — the whole point of
adopt is that WOM uploads nothing. So the store could never reach upload tier 2, and the
batch adopt could never be unblocked. A HEAD-only operation was gated on a 5 GiB PUT proof
it structurally could never produce.

## The fix — an adopt-specific boolean ladder

Adopt is now **decoupled** from the upload 3-tier ladder. Two new pure functions carry the
adopt rule, adjacent to the upload tier functions:

- `object_storage_adopt_requested_tier(*, object_count)` — binary: a single tiny-first adopt
  is tier 1; any batch (> 1 object) is tier 2.
- `object_storage_adopt_proven_tier(root, *, provider_kind, store_ref)` — returns 1 once the
  store has **>= 1 prior verified-adopt receipt**, else 0.

The gate arithmetic is unchanged (`requested - 1 > proven` blocks): a single adopt
(requested 1) needs proven >= 0 and is always permitted (tiny-first preserved); a batch
(requested 2) needs proven >= 1 — i.e. **exactly one prior verified tiny-first adopt unlocks
a batch adopt of any size N**. There is no adopt 5 GiB rule and no adopt distinct-object
minimum, because adopt moves zero bytes and each object still runs its own per-object
HEAD-verify self-limit; a byte- or count-based middle tier for adopt would bound nothing
real.

The operator remedy is one command:

```
object-storage-adopt-existing --only <one-sha> --approve   # a verified tiny-first adopt
# then re-run the full --key-map batch of 19,054 objects — it now proceeds.
```

## The proof is adopt-specific and positive

`object_storage_adopt_proven_tier` reads ONLY `*.object-storage-upload.json` execution
receipts (never manifest `declared_uploaded`/`wom_uploaded` locations), and counts a receipt
as a verified-adopt proof only when it carries a verified adopt marker:
`adopt_verification in {"presence_size", "content_hash"}` (the sole assignment site is
`_object_storage_write_adopt_receipt`) **and** `result_status == "skipped_remote_same"`.

`result_status` and `bytes` alone do not distinguish adopt from upload — an upload that
skips on already-matching bytes ALSO writes `skipped_remote_same` + `bytes_uploaded: 0`. The
`adopt_verification` marker is the load-bearing discriminator; the `skipped_remote_same`
cross-check is defense in depth.

## Safety envelope (unchanged, and re-tested)

- **Per-object HEAD-verify self-limit intact.** Even with the gate open, an object that is
  absent or size-mismatched at its key yields `absent_or_size_mismatch` and is NOT adopted.
  A wrong `--key-map` still self-limits to zero false adopts.
- **Declared/unverified adopt never counts and never gates a PUT.** `--accept-unverified-adopt`
  writes a manifest `declared_uploaded` location but NO execution receipt, so it can never
  enter the adopt proof, never opens a batch, and never lets a later upload skip its PUT.
- **The upload tier ladder is byte-identical.** `object_storage_requested_tier` /
  `object_storage_proven_tier` and the upload gate (which keeps `tiered_gate_unmet`) are
  untouched. An upload receipt does NOT unblock adopt, and an adopt receipt
  (`bytes_uploaded: 0`) can never reach upload tier 2, so no reverse leak unlocks a PUT.
- **`--content-hash-verify` and `--max-objects` unchanged.** The per-object cryptographic
  opt-in still downloads-and-rehashes only when asked; `--max-objects` still refuses a run
  whose resolved plan exceeds the bound.

## Token rename

The adopt blocker is renamed `tiered_gate_unmet` → `adopt_tiny_first_unmet`, with a message
that names the exact remedy and states that an upload / 5 GiB proof does not help. The upload
gate keeps `tiered_gate_unmet` byte-identical, so tests and operators can assert the two
gates independently. This supersedes the F5 decision log
(`docs/archive-infra-decision-log-2026-07-04-v03166-key-strategy-adopt.md`), which is
historical and not edited; the v0.3.174 decision log records the supersession.

## Compatibility

Additive; **no migration required**. Existing receipts and manifests are unaffected. The
full test suite, `check_public_privacy`, and `check_release_readiness` stay green.

## Upgrade

See `UPGRADE.md`. All changes are additive; no migration is required. The only behavioral
change is that a batch verified adopt now proceeds after one verified tiny-first adopt (it
previously required an upload 5 GiB tier proof it could not produce), and the adopt blocker
token is now `adopt_tiny_first_unmet`.
