# v0.3.184 - zet frontmatter viewer contract

v0.3.184 responds to separate real-use feedback that canonical zet files can
look broken when raw YAML frontmatter fences are shown to a beginner who expects
a document.

This release is additive. It does not change the canonical storage format.

## Human document view

`archive read-zettel` now accepts:

```powershell
archive read-zettel <archive-root> --zettel-id <id> --section document
```

and:

```powershell
archive read-zettel <archive-root> --path zettels/example.md --section document
```

In text mode, `document` prints only the zet body. It does not print the raw
frontmatter fences, and it does not print the normal CLI path/id/status header.

In JSON mode, `document` includes the body while hiding full frontmatter details.

## Viewer contract flags

`read-zettel` output now includes:

- `viewer_mode`
- `frontmatter_hidden`
- `raw_frontmatter_delimiters_echoed`

For `--section document`, these are:

```json
{
  "viewer_mode": "human_document",
  "frontmatter_hidden": true,
  "raw_frontmatter_delimiters_echoed": false
}
```

These fields make the intended rendering contract explicit for simple viewers,
editors, and AI runtimes: frontmatter is storage metadata, not the document body.

## AI response guidance

`archive ai-response-concept-guide --topic zet_markdown_style --dry-run` now
also reports the frontmatter viewer rule. AI runtimes can discover that:

- frontmatter is canonical storage metadata;
- human document views should hide or fold it;
- `read-zettel --section document` is the body-first read path.

## Frontmatter remains canonical metadata

The YAML frontmatter remains part of canonical WOM zet storage. It carries ids,
facets, provenance, visibility, edges, and receipt references. Tools should hide
or fold it in human reading surfaces, not delete it.
