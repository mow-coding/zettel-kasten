# v0.3.2 - Frontmatter Migration, Redaction Hardening, And The Local Capture Spine

Release date: 2026-06-11

This release closes the system's first complete local loop: stage a file, preserve
it as a content-addressed objet, draft and mint a zet that references it, find it
again through typed edges and facet views, and verify the staged folder is safe to
clean up. It also ships the frontmatter v0.3 compatibility migration and hardens
redacted-zettel suppression across every read surface. The v0.3.1 frontmatter
schema itself does not change.

## Added

### Frontmatter compatibility migration

- CLI `archive migrate <archive-root> --target frontmatter-v0.3 --dry-run --format json`.
- CLI `archive migrate <archive-root> --target frontmatter-v0.3 --approve --format json`.
- Doctor compatibility output showing release/schema compatibility.
- Doctor migration hints for legacy frontmatter failures.
- v0.3 zettel-rules guidance for required nested `provenance` and `visibility` fields.

### Local objet capture (approval-gated write)

- CLI `archive objet-capture <archive-root> --selection <manifest.json> --dry-run | --approve --reviewed-by <actor>`.
- Captures explicitly approved staged files into the local content-addressed store
  at `objects/sha256/<2-hex>/<64-hex>`, appends one manifest record per objet to
  `objects/manifests/files.jsonl`, and always writes a capture receipt under
  `receipts/objet-capture/`.
- Selection manifests carry a per-item `approved_object_id` content commitment and
  a persisted source-intake plan binding; the run refuses archives without an
  explicit sandbox marker (`.wom-sandbox` file or top-level `environment: sandbox`).
- Capture is lossless (stored bytes are re-hashed and verified), idempotent
  (re-running skips already-present objets), TOCTOU-resistant (single-descriptor
  hashing and copying), and never deletes or modifies staged originals.

### Staged-folder cleanup verifier (report-only)

- CLI `archive staged-cleanup-check <archive-root> --staged <folder> --dry-run`.
- Reports, per staged file, whether it is preserved (store copy present, bytes
  re-verified, canonical manifest record present), explicitly deferred, or not
  preserved, plus how many zets reference each objet.
- `safe_to_cleanup` is true only when nothing is unpreserved or unsafe, and the
  scan fails closed on any tree it cannot fully enumerate. The command never
  deletes anything; cleanup remains a manual human action.

### Typed-edge retrieval and facet views (read-only)

- The disposable search index now also indexes typed edges and zettel facets.
- CLI `archive related-zets <archive-root> --zettel-id <id> [--depth 1-3] [--edge-type t]`:
  bidirectional traversal over typed edges, so backlinks ("what supersedes or
  derives from this zet") are answerable for the first time.
- CLI `archive view-zets <archive-root> --view-id <id> | --facet key=value ...`:
  executes the facet filters already declared in `views/*.yml` (top-level or
  `saved_views`) or ad-hoc ANDed facet filters. Unsupported filter keys block
  instead of silently broadening the result.

### Hygiene and platform fixes

- Report-only artifact hygiene checker (`wom-kit/tools/check_artifact_hygiene.py`)
  with a six-class file-lifecycle baseline (`wom-kit/docs/artifact-hygiene.md`).
- Generated-archive `.gitignore` now excludes the `objects/sha256/` byte store.
- Setup-script fixes: Windows drive-path recognition and a canonicalized
  repository-root guard in `setup-unix.sh`; robust bootstrap-test output decoding.

## Privacy

Redacted-zettel suppression is now enforced consistently across every read
surface: search results, the on-disk search index (a redacted zettel's title,
body, frontmatter, edges, and facets are never indexed), `list-zettels`,
`read-zettel`, block-header previews, projection previews, and the new
related-zets/view-zets queries (a redacted zettel is never returned as a
neighbor, never accepted as a traversal subject, and never matches a facet
filter). Regression tests cover each surface.

## Behavior

The migration scans archive-contained Markdown zettels under `inbox/` and
`zettels/`. The dry-run returns per-field planned changes and writes nothing.
The approve path rewrites only reviewed zettel frontmatter and is idempotent.
Clean object-shaped `provenance.source` values are preserved by relocating their
canonical JSON representation into `source_refs`; ambiguous, unsafe, secret-like,
provider URL, or local absolute path values block for manual review.

Capture, cleanup-check, related-zets, and view-zets follow the same house rules
as existing commands: dry-run first, approval-gated writes, JSON output, explicit
blockers, and no provider or network calls.

## Safety Boundary

This release does not redesign the v0.3.1 schema, touch private live archives,
upload or sync providers, delete staged originals or cleanup targets, create ZET
transport, mutate trust, or add any MCP write tool. The objet store write path is
restricted to explicitly marked sandbox archives.

## Upgrade

Run the migration dry-run before approving any private archive rewrite:

```text
archive migrate <archive-root> --target frontmatter-v0.3 --dry-run --format json
```

Apply only after reviewing the plan on a backup or sandbox copy:

```text
archive migrate <archive-root> --target frontmatter-v0.3 --approve --format json
```

Rebuild the search index once to pick up edges and facets:

```text
archive index <archive-root>
```

See the [Upgrade Guide](https://github.com/mow-coding/zettel-kasten/blob/main/UPGRADE.md) and [Changelog](https://github.com/mow-coding/zettel-kasten/blob/main/CHANGELOG.md).
