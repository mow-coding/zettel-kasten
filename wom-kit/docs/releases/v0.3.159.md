# v0.3.159 - Paired Transcript Intake And BOM-Aware Derive-Text Encoding

v0.3.159 lets ONE reviewed selection manifest approve both a staged original
(for example a call recording) and its already-extracted transcript, and makes
derived-text capture decode BOM-marked UTF-8/UTF-16 input deterministically
instead of refusing everything that is not already BOM-less UTF-8. Motivating
case: Samsung Voice Recorder exports an `.m4a` plus a UTF-16-LE CRLF `.txt`
side by side; previously the transcript needed a second, separately-approved
`derive-text capture` run and the UTF-16 file was refused outright.

## What Changed

- Selection-manifest items MAY carry a `derived_text` sub-object inside the
  hashed manifest: `staged_text_path` (archive-relative), `approved_text_sha256`
  (over the RAW file bytes; the always-blocking approval commitment mirroring
  `approved_object_id`), `derivation_kind`, `tool_name`, `tool_version`,
  `review_status`, and optional `model_name`/`model_version`/`confidence`/
  `language`/`born_digital`. `selection_manifest_sha256` recomputation covers
  the pair, so a swapped transcript blocks with
  `approved_text_content_mismatch`.
- Paired manifests use the NEW envelope: `action:
  local_objet_capture_with_derived_text_approved`, `schema:
  wom-kit/b4-selection/v0.3`. The `action` exact-match is the only lever old
  kits enforce, so v0.3.158-and-earlier kits refuse paired manifests with
  `selection_action_invalid` instead of capturing the original and silently
  dropping the approved derived half. Plain manifests keep the old action and
  `wom-kit/b4-selection/v0.2`; the reader now validates the `schema` field for
  the first time (`selection_schema_invalid`), fail-closed in both directions
  (derived items under the plain action, or a paired action without any
  derived item, refuse with `selection_action_invalid`).
- `objet-capture-selection` gains `--derived-text-staged-path` plus the
  derive-text metadata flags; generation runs the full staged_path-parity
  confinement chain on the transcript path, hashes the raw bytes with an
  O_NOFOLLOW-fd + fstat read, and validates the metadata so a bad pair blocks
  before a human ever approves it.
- `staged_text_path` confinement has full parity with `staged_path` at
  generation AND capture time: normalize/resolve/containment,
  internal-prefix block (a transcript "source" cannot point into `receipts/`
  or `objects/`, including `objects/derived-text/sha256` itself), reserved
  device names, enablement-aware never-touch, per-component symlink/junction
  chain checks, and `duplicate_selection_target` when one file appears as both
  an original and a transcript source (or twice as a transcript source).
- `objet-capture --approve` runs two phases inside the same
  `_ObjetCaptureManifestLock`: phase 1 publishes originals unchanged; the
  objet manifest handle is then flushed, fsynced, and closed (ordering is
  load-bearing: a derived record referencing a nonexistent object_id after a
  crash is impossible by construction); phase 2 registers each derived half
  through the derive-text store bound to the phase-1 `object_id`, with the
  derive-text lock nested inside (lock order ObjetCapture -> DerivedText,
  never the reverse).
- Outcomes: a blocked original never reads its transcript
  (`blocked_by_original`); a phase-2 exception converts to the item-level
  blocker `derived_text_registration_failed` without aborting the run;
  original-durable + derived-blocked reports additive `status_class: partial`
  (item and run, v0.3.151 taxonomy) with `ok: false` and `next_safe_actions`
  naming the two safe completion routes (re-run the SAME selection —
  idempotent repair — or standalone `derive-text capture` with the minted
  object id). Re-runs repair partials: the original half skips, the derived
  half retries.
- Receipt topology: one objet-capture run receipt (schema
  `wom-kit/objet-capture-receipt/v0.3`) whose items carry the full
  `derived_text` sub-result including the derived receipt path, plus the
  normal `receipts/derived-text-capture/` receipt per pair (schema
  `wom-kit/derived-text-capture-receipt/v0.2`) carrying `paired_with:
  {selection_manifest_id, selection_manifest_sha256, item_id}`. Summary gains
  `derived_text_written`/`derived_text_skipped`/`derived_text_blocked`
  (approve) and `would_register_derived_text`/`derived_text_blocked`
  (dry-run) only on paired runs.
- Deterministic BOM-only encoding ladder for ALL derive-text capture paths
  (standalone single-file, `--from-manifest` batch, paired): utf-8-sig BOM ->
  decode utf-8-sig; UTF-32 LE/BE BOM -> block
  `text_file_bom_encoding_unsupported` (checked BEFORE the 2-byte UTF-16
  check: UTF-32-LE's BOM prefix-collides with UTF-16-LE's); UTF-16 LE/BE BOM
  -> strict decode; no BOM -> strict UTF-8 (exactly the old acceptance).
  BOM-marked-but-undecodable input blocks with
  `text_file_bom_encoding_undecodable` + `detected_bom`; decoded text
  containing U+0000 blocks with `text_file_contains_nul`; BOM-less non-UTF-8
  keeps `text_file_not_utf8` with an extended hint naming the auto-handled
  encodings. No chardet, no guessing, `errors="strict"` everywhere, and NO
  CRLF or other content normalization — stored text is decode -> strip
  leading BOM -> encode UTF-8, line endings preserved.
- Hash identity (pinned): `text_sha256`, `text_logical_key`,
  `derived_text_id`, `size_bytes`, `mime`, and lossless verification are all
  computed over the STORED normalized UTF-8 bytes; `source_text_encoding` and
  `source_text_sha256` (raw bytes) are recorded in record provenance and
  receipts. Two encodings of identical text collapse to one
  `derived_text_id`; the second registration is `skip_already_present` and
  keeps the first record's `source_text_encoding`, while the second run's
  receipt still records the second source's encoding/hash.
- Added `DERIVED_TEXT_MAX_SOURCE_BYTES = 64 MiB` (blocker
  `text_file_too_large`, checked on the fstat size before any read; applies
  to standalone and paired) and upgraded the standalone `--text-file` reader
  to an O_NOFOLLOW-fd + fstat read, closing its lstat->read TOCTOU window.
- Discoverability: source-intake next safe actions and the
  project-intake-item-plan `command_guidance` (with an `audio/*` pairing
  hint) now name the paired flow; the capability matrix rows are updated.

## Safety Boundary

- The pair adds NO MCP-reachable write path: objet-capture and derive-text
  capture remain CLI-only, and v0.3.158 kept the enablement flow CLI-only.
  Refusals stay blocker-only: no staged text content and no filenames beyond
  the existing archive-relative staged-path echo policy; encoding hints are
  static, blocker-keyed strings.
- The v0.3.158 enablement gate covers the paired derived half only because
  the pair runs INSIDE objet-capture (gate by containment); standalone
  derive-text capture remains ungated by design (`gate_scope` unchanged).
- Deliberate confinement asymmetry, documented not removed: standalone
  `--text-file` accepts arbitrary absolute local paths (symlink/dir/special
  checks plus the new fd read and size cap only); the manifest-carried
  `staged_text_path` is stricter because it is a persisted approval record.
- Paired intake does NOT preserve raw transcript bytes: stored text is
  transcoded UTF-8, so `staged-cleanup-check` will correctly report the
  staged `.txt` as `not_preserved` (preservation requires an `objects/sha256`
  copy hashing to the raw digest). When raw-byte preservation matters, use
  the deferred list or a separate objet capture of the `.txt`.
- The Windows O_NOFOLLOW residual on the fd readers is the same documented
  policy as before (lstat pre-check plus per-component chain checks remain
  the guard there).

## Honest Non-Additivity

utf-8-sig files were ACCEPTED before this release — the BOM survived the
UTF-8 validation and the raw bytes were stored WITH the BOM — so BOM
stripping changes `text_sha256`/`derived_text_id` for the same input. A
post-upgrade re-run of the same utf-8-sig input creates a SECOND record
instead of `skip_already_present`. See UPGRADE.md.

## Upgrade Notes

No archive migration is required and no JSON fields are renamed. Plain
(non-paired) selection manifests keep the old action string and the v0.2
schema and behave byte-identically; their summaries carry no derived
counters. JSON consumers of objet-capture results see additive fields only
(`status_class`, `derived_text` sub-results, derived summary counters on
paired runs, `next_safe_actions` on partial). The derived-text capture
receipt schema id changed (v0.1 -> v0.2) and the objet-capture receipt schema
id changed (v0.2 -> v0.3); consumers that exact-match schema ids need the new
values. The utf-8-sig hash-identity change above is the only non-additive
behavior change.
