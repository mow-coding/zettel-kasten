# v0.3.107 - Source Map Scale Fix

Date: 2026-06-18

## What Changed

This release fixes a large-manifest startup hang in:

- `archive notion-objet-source-map-link-plan`
- `archive notion-objet-import-clue-audit`

The source-map planner now preloads `objects/manifests/files.jsonl` once and
reuses a local object-id index while building material-link candidates. It no
longer resolves every manifest object through repeated full manifest scans.

The local candidate existence check in this planner also avoids repeated
archive-root resolution for already normalized archive-relative paths.

## Why

On small fake archives, the old path completed quickly. On a larger real-use
shape with about 19k object manifest rows and 300 imported zettels, the same
commands could appear to hang at startup and produce zero output before a user
timeout.

The issue was algorithmic, not a provider, stdin, source-map size, or Python
environment problem.

## Verification

Added a regression test that runs the source-map planner and import clue audit
against a scaled manifest fixture while failing the test if `resolve_objet_ref`
is called per manifest object.

The patched command path was also checked against a large local archive shape:

- object manifest rows: 19,055
- zettel sources: 303
- source-map rows: 121
- ledger rows: 120

Both affected commands completed in seconds and returned JSON summaries.

## Safety Boundary

This release is still read-only. It does not:

- read zettel body text,
- read object bytes,
- call providers,
- start OAuth,
- echo provider URLs,
- write edges,
- write receipts,
- rewrite zettel bodies,
- create presigned URLs.

The fix only changes the local planning algorithm and test coverage.
