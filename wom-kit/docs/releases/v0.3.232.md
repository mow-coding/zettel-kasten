# v0.3.232 - Explicit Abstract Publication Gate

v0.3.232 closes the forward-creation gap found after the v0.3.231 readiness
gate: new canonical memory can no longer be published without an explicit
compact first read. The change is additive and requires no archive migration.

## What Changed

- `create-draft` remains permissive when `abstract` is absent and reports a
  content-free not-ready `first_read_check`.
- `mint-zet`, `mint-zettel`, and legacy `promote` require an explicit
  `frontmatter.abstract` that is normalized to one line, at most 360
  characters, and free of private locators or secret-like material.
- `gist`, `summary`, `description`, and `overview` remain compatibility first
  reads but cannot authorize canonical publication.
- Real mint and promotion compare both the complete draft SHA-256 and abstract
  SHA-256 after dry-run, using one captured byte snapshot for canonical text,
  the mint snapshot, and receipt evidence. Any draft-byte drift or a missing
  or invalid abstract blocks before any write.
- Mint and promotion previews, results, and receipts carry a text-free
  `first_read_check`; the optional mint-receipt schema field preserves older
  receipt compatibility.

## Upgrade And Compatibility

Existing canonical zets and receipts remain valid. No automatic abstract is
generated and no old material is rewritten. Add and review an explicit
abstract only when a draft is ready to become canonical, then rerun the normal
mint dry-run.

```text
archive mint-zet <archive-root> --path inbox/<draft>.md --dry-run --format json
```

Proceed only when `first_read_check.ready_for_publication` is `true`.

## Safety And Privacy

- The publication check receives frontmatter only and does not read the body.
- The check evidence stores no abstract text, title, body, path, provider
  value, or secret.
- Dry-run never grants approval; the ordinary reviewer and approval gates
  still apply.
- Passing does not prove semantic truth, completeness, freshness, or model
  consumption.
