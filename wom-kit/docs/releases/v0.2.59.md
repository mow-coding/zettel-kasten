# v0.2.59 - ZET Transport Threat Model And Would-Transport Plan

Release date: 2026-06-02

v0.2.59 adds a read-only threat model and planning preview for future ZET transport.

## Added

- CLI `archive zet-transport-plan <archive-root> --record <path> --method <key-sharing|radio-frequency|mirroring> --dry-run --format json`.
- MCP `zet_transport_would_plan`.
- Service `zet_transport_would_plan`.
- Documentation: [ZET Transport Threat Model](https://github.com/mow-coding/zettel-kasten/blob/main/wom-kit/docs/zet-transport-threat-model.md).
- Work log: [ZET Transport Threat Model And Would-Plan Work Log](https://github.com/mow-coding/zettel-kasten/blob/main/wom-kit/plans/work-log-2026-06-02-zet-transport-threat-model-would-plan.md).

## Behavior

- The planner is dry-run only.
- It writes no files and returns `would_change: []`.
- It reads one local archive-contained shared update record JSON only.
- It reuses the v0.2.56 single-record shared update review preview policy before producing any plan.
- It returns method-specific planning risks and future controls for `key-sharing`, `radio-frequency`, or `mirroring`.
- It returns no body text, local absolute paths, provider URLs, tokens, secrets, keys, signatures, endpoints, radio-frequency handles, mirror payloads, queue jobs, workers, receipts, or delivery objects.

## Safety Boundary

This release does not implement real ZET transport, key creation, key-sharing registry, radio-frequency access creation, mirroring delivery, shared-update review writes, receiver-side renewal writes, neighbor feed update, recommendation execution, trust/import/acceptance/anchor, attestation/signature writes, provider sync, WordPress publishing, projection writes or receipts, queues, workers, payments, staking, consensus, blockchain, model training, backpropagation, or full-auto behavior.

MCP exposes only the read-only dry-run tool. It does not expose apply/write/send/deliver/publish/transport/import/trust/attest/sign/anchor/key/radio-frequency/mirror sibling tools.

## Upgrade
