# v0.3.213 - local sovereignty and backup authority

v0.3.213 gives AI operators and humans one machine-readable answer for which
WOM copy is canonical and what GitHub, object storage, and external databases
back up. It is additive and requires no archive migration.

## What Changed

- Added read-only CLI `local-sovereignty`, with `storage-authority` and
  `backup-authority` aliases.
- Added the same `storage_authority` model to `runtime-context`,
  `ai-start-here`, `recovery-plan`, and `upgrade-check` recovery output.
- Declared local reviewed WOM files and registered local objet bytes as the
  canonical working/recovery state.
- Declared GitHub metadata/version backup, object-storage objet-byte backup,
  and external-DB map backup/replica roles.
- Added offline capability, conflict precedence, recovery order, disposable
  index/map, and backup-evidence boundaries.
- Corrected older source-map and closed-sharing descriptions so storage role
  no longer implies authority.

## Honest Evidence Boundary

A local Git commit alone is not remote-backup proof. A provider-verified
`wom_uploaded` object-storage receipt can prove its implemented boundary, while
`declared_uploaded` cannot. Generic GitHub and external-DB backup completion
receipts are still not implemented and are reported as gaps.

## Compatibility And Safety

`ai-start-here` schema advances to v0.2 because it adds the authority model.
Existing archives and zets require no migration. The new command reads no zet
body or objet byte, calls no provider or network, reads no secret, and writes
no file, receipt, index, map, or external backup.
