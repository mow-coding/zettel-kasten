# v0.3.160 - AI Intake Protocol And Store Guards

v0.3.160 closes the remaining non-capture gaps from the 2026-07-03 client
voice-memo letter (frictions 1/3/4), applies the dev-team intake layout ruling
D2, and fixes survey finding F-D. Motivating case: every runtime-visible
surface gated source intake on "before DRAFTING", so an agent could still
physically copy a client's multi-hundred-GB folder into the archive or a raw
`objets/` folder without any gate, a `git init` at a workspace root above an
objet byte store would start tracking raw originals, and the shipped
operator-feedback commands were undiscoverable and had no schema files.

## What Changed

- AI Intake Protocol (friction 1): the personal/family/company AGENTS.md
  templates, the fake-life-archive example AGENTS.md, the runtime SKILL.md,
  and `docs/wom-ai-runtime-skill-plugin-layer.md` now say, normatively: BEFORE
  physically copying any local file into the archive or an objet store, run
  `archive source-intake <archive-root> --dry-run --local-path <file>`; stage
  inside the archive root (recommended
  `staging/incoming/<YYYY-MM-DD>/<project_slug>/`); follow
  `next_safe_actions`; capture only through the reviewed
  `objet-capture-selection` -> `objet-capture` approval chain (optionally
  pairing a vendor transcript via `--derived-text-staged-path`); real
  archives additionally need an owner-approved `objet-capture-enable` record;
  and bulk external stores route through `prehashed-objet-ledger` plus
  `object-storage-upload-evidence` instead of per-file copies. Capture
  authority comes ONLY from the reviewed selection plus the approved capture
  (plus enablement) — never from an intake plan.
- Workspace-root objets git guard (friction 3): doctor gains two additive
  read-only warnings. `archive_objets_layout_noncanonical` fires whenever a
  raw in-root `objets/` folder exists and points at the migration guide; it is
  deliberately NOT silenced by gitignoring the folder, because once `/objets/`
  is ignored, unmigrated originals silently drop out of the git-push backup
  path. `workspace_objet_store_git_exposure` fires when an objet BYTE store
  (in-root `objets/` or the sibling `<root-name>-objets` store) sits inside a
  git working tree whose ignore rules do not plainly exclude it; the message
  names the store's actual directory name (the `objets/` pattern does not
  match a sibling `<root-name>-objets` store) and only the basename — never a
  local absolute path. The recommended line is anchored (`/<name>/`) when the
  store is a direct child of the repository root and unanchored (`<name>/`)
  when it sits deeper, because an anchored repo-root line would not match a
  nested store in git; anchored `.gitignore` lines are likewise only accepted
  as silencing when they sit in the store's own parent directory.
- Guard scope decisions (documented, not accidental): the
  archive-is-its-own-git-repo case is the supported workflow and stays silent
  (the `.gitignore` completeness check owns that signal); worktree/submodule
  `.git` pointer files count as working-tree roots; bare repositories and
  stores that are their own git repos never warn. "Own repo" requires a VALID
  `.git` marker (a `.git` dir with `HEAD`, or a `gitdir:` pointer file whose
  target exists with `HEAD`): a broken marker — empty `.git` dir, dangling
  pointer — is ignored by real git, so an enclosing repository tracks the
  store's contents anyway, and the guard treats broken as ambiguous and still
  warns. The check is a cheap filesystem heuristic (bounded walk-up, a few
  small `.gitignore` reads, no `git` subprocess, no store enumeration), so
  the warning is worded "may be tracked".
- Recommended `.gitignore` defaults gain the anchored `/objets/` line
  (`init`, `repair-gitignore`, doctor completeness check,
  `tools/check_artifact_hygiene.py`, docs listing, and the fake-life-archive
  fixture). Anchored on purpose: `staging/incoming/<date>/<client-tree>/objets/`
  and other nested `objets/` folders keep their normal handling.
- Intake layout ruling D2 (dev-team decision, Kim-delegated 2026-07-03), one
  consistent statement across runtime and docs: canonical capture intake
  stages INSIDE the archive root at `staging/incoming/<YYYY-MM-DD>/` (date
  layer recommended, not required; capture requires archive-relative staged
  paths and v0.3.158 enablement makes this the real-archive path); the
  sibling `-objets` store holds bulk external originals under never-touch
  protection with prehashed-ledger evidence; a raw in-root `objets/` folder
  is non-canonical, with an in-place migration guide (source-intake ->
  source-intake-record -> objet-capture-selection --staged-path objets/<file>
  -> objet-capture -> staged-cleanup-check) in `docs/artifact-hygiene.md`.
- The runtime now matches the ruling instead of contradicting it:
  `project_intake_staging_convention` accepts in-archive `staging/incoming/`
  staging as `matches_recommended_shape` (additive `matched_shape`,
  `recommended_in_archive_shape`, and `in_archive_staging_supports_capture`
  fields), project-intake-plan/unpack-queue warnings name both accepted
  shapes, and project-intake-staging-guide recommends the in-archive shape
  for capture intake while keeping the sibling shape for bulk external
  originals.
- Operator-feedback discoverability (friction 4) and schema files (F-D):
  `archive operator-feedback-plan` joins
  `runtime_context_recommended_first_commands` (appended entry),
  `ai_runtime_order` (new step 7 `plan_operator_feedback`), and
  `RUNTIME_CONTEXT_SAFE_ACTIONS`; `wom-kit/schemas/operator-feedback.schema.json`
  and `wom-kit/schemas/operator-feedback-receipt.schema.json` now ship,
  matching the record/receipt shapes exactly (schema-id strings unchanged,
  nullable fields typed `["string", "null"]`, constraints kept inside the
  in-house validator subset: type/required/properties/items/enum/const).

## Safety Boundary

- Both new doctor warnings are read-only and fail-quiet; no new write path,
  no MCP tool, no provider call, and no `git` subprocess is added.
- The exposure warning echoes the store basename only. Doctor already prints
  the inspected archive root, but paths outside the archive root are not
  echoed by the new checks.
- The gitignore additions are additive lines only; `repair-gitignore` keeps
  its approval gate. gitignore does not untrack already-committed files —
  that cleanup stays a human `git rm --cached` decision after review.
- The new AGENTS/SKILL wording adds no new capability: it routes agents into
  the existing v0.3.157-159 gate chain.

## Honest Strict-Gate Impact

Two tiers. First, EVERY archive created before v0.3.160 — even a clean one
with no `objets/` folder — newly trips the pre-existing
`local_profile_gitignore_incomplete` warning because `/objets/` joined the
recommended `.gitignore` defaults; that alone fails `validate` (which fails
on warnings by default) and `doctor --strict` until one
`repair-gitignore --approve` run adds the line. Second, the new warnings can
fail the same gates plus `runtime-context --strict` for archives that
currently keep originals in an in-root `objets/` folder or an exposed store —
including the client archive this release is meant to help — until the
migration guide is followed or the exposure is gitignored. This is a
deliberate, documented decision (unmigrated originals in an ignored folder
are invisible to git backup), not an accident. See UPGRADE.md.

## Still Future

- No automatic migration, move, or cleanup of an in-root `objets/` folder;
  the guide is manual and receipt-gated, and `staged-cleanup-check` stays the
  only deletion-safety verdict.
- No `git check-ignore` subprocess integration; the exposure heuristic reads
  plain `.gitignore` lines only and says "may be tracked".
- No provider upload/sync path for the sibling store beyond the existing
  evidence commands.

## Upgrade Notes

No archive migration is required and no JSON fields are renamed or removed.
JSON consumers see additive fields only (`matched_shape`,
`recommended_in_archive_shape`, `in_archive_staging_supports_capture`, one
appended `recommended_first_commands` entry, one appended `ai_runtime_order`
step, and one new safe action `run operator-feedback-plan dry-run` INSERTED
mid-list next to the other read-only dry-run actions — position 3 of 8, so a
consumer pinning `available_safe_actions` by index must reindex). The two new
doctor warning ids and the `/objets/` gitignore addition are additive, BUT
the strict gates are affected in two tiers: every pre-v0.3.160 archive newly
trips `local_profile_gitignore_incomplete` (missing `/objets/`) until
`repair-gitignore --approve` runs, and archives with an in-root `objets/`
store or an exposed byte store additionally carry the two new warnings until
migration completes — see UPGRADE.md for the exact sequence.
