# v0.3.117 - AI Operational Context Rehydration

Date: 2026-06-20

## What Changed

This release adds a bounded AI-facing operating-memory layer for WOM archives.

An archive can now keep a reviewed mission/state handoff record at:

```text
ops/operational-context.yml
```

`archive runtime-context <archive-root> --format json` now includes
`operational_context`, including a compact `session_start_injection` with the
current mission, scope, phase, completed/in-progress/next/blocked items,
gotchas, and decisions.

This is meant for session reset and context-compression recovery. It helps an AI
runtime remember what the archive is currently trying to do before it starts
reading broad zettel bodies or source artifacts.

## New Command

Read the current record:

```powershell
archive operational-context <archive-root> --dry-run --format json
```

Validate a staged candidate:

```powershell
archive operational-context <archive-root> --record workbench/operational-context.next.yml --dry-run --format json
```

Approve a reviewed update:

```powershell
archive operational-context <archive-root> --record workbench/operational-context.next.yml --approve --reviewed-by <actor> --format json
```

Approved updates replace `ops/operational-context.yml` and write a receipt under
`receipts/operational-context/`.

## Safety Boundary

Operational context is not a replacement for zets, receipts, manifests, or
source maps. It is a small handoff record for AI runtimes.

This release does not:

- scan broad archive bodies,
- auto-resolve contradictory old notes,
- call providers,
- read secrets,
- store provider URLs or local absolute paths,
- expose an MCP write tool,
- add UI behavior,
- add ZET transport,
- change mint semantics.

Candidate values that contain provider URLs, local path hints, email-like
account labels, tokens, or secret-like values are blocked before write.
