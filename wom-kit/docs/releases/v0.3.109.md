# v0.3.109 - Migration Rollback Safety

Date: 2026-06-18

## What Changed

This release adds the safe, receipt-backed part of migration rollback:

- `archive migrate --target link-types-v0.3 --revert --dry-run`
- `archive migrate --target link-types-v0.3 --revert --approve`

Forward `link-types-v0.3` approval now writes a migration receipt under
`receipts/migrations/`. The revert reads that receipt and removes only
recommended connection edge type records that are:

- listed in the receipt as appended,
- unused by current zettel edges,
- unchanged from the base WOM-kit type template.

It also recognizes `archive migrate --target frontmatter-v0.3 --revert`, but
that path fails closed with `snapshot_receipt_required`.

## Why

v0.3.108 added receipt-based rollback for approved edge writes. The remaining
rollback request was migration rollback. Link type migration can be safely
reverted when the added vocabulary has not been used yet. Frontmatter migration
is different: restoring old frontmatter without a prior snapshot receipt can
lose or guess user data.

## Safety Rules

`link-types-v0.3 --revert` blocks if:

- no matching migration receipt exists,
- a receipt-listed edge type is used by any current zettel edge,
- a candidate edge type record differs from the base template.

`frontmatter-v0.3 --revert` blocks until a future migration snapshot receipt
exists.

## Verification

Added regression coverage for:

- successful `link-types-v0.3 --revert` dry-run and approval,
- blocking link-type revert while a candidate type is used by a current edge,
- blocking `frontmatter-v0.3 --revert` with `snapshot_receipt_required`.

## Safety Boundary

This release does not:

- call providers,
- read source exports,
- write edges,
- delete edge receipts,
- remove unreceipted link type records,
- infer relationship meanings,
- guess old frontmatter,
- restore frontmatter without a snapshot receipt.
