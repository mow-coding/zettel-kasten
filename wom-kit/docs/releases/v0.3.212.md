# v0.3.212 - compact strict continuations

v0.3.212 reduces repeated metadata during long strict `zet-catalog` passes
without reducing node coverage. It is additive and requires no archive
migration.

## What Changed

- Added CLI `--response-profile continuation` and MCP
  `response_profile: "continuation"` for nonzero strict cursors.
- Kept cursor zero full so the host receives scope-wide abstract, identity,
  order, scan, and closed-action diagnostics once.
- Omitted those repeated sections on compact continuation pages while keeping
  items, coverage/readiness, snapshot, continuation token, chain evidence,
  session consistency, page/response measurements, warnings, and privacy
  guards.
- Added a machine-readable response-profile contract and explicit proof that
  profile changes are not bound into the strict continuation token.
- Extended the synthetic benchmark with full/continuation page counts and
  cumulative compact service-result estimates.

## Scale Evidence

On equivalent 10,000-node seeded strict-reading runs, 264 full pages measured
1,785,893 cumulative service-result tokens. Keeping the first page full and
using 263 compact continuation pages measured 1,671,758: 114,135 fewer, or
6.39%. Both runs returned all 10,000 unique nodes and completed all three
readiness signals with no item or response budget overrun.

## Compatibility And Safety

Catalog output schema advances to v0.8. Strict continuation schema stays v0.3
because response profile does not change selected items, order, cursor, or
chain identity. Existing callers receive full responses unless they opt in.

The compact profile is rejected at cursor zero and outside strict coverage
mode. It is response shaping only: no zet body or objet byte is read, no model
or provider is called, and no archive, map, receipt, goal, or loop state is
written.
