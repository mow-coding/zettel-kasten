# v0.3.251 - Honest Local Backup Evidence Status

v0.3.251 closes the gap between WOM's local-sovereignty authority model and
the backup evidence a current archive can honestly show without a remote call.

## What Changed

- New read-only CLI `backup-evidence`, with aliases `backup-status` and
  `storage-backup-evidence`.
- The command reports GitHub, object-storage, and external-database backup
  lanes separately in machine-readable JSON.
- Object-storage coverage counts only `wom_uploaded` locations linked to valid,
  matching provider-confirmed execution receipts.
- Partial and full coverage are explicitly limited to the receipts' recorded
  time; current remote availability remains unchecked.
- Missing, forged, malformed, duplicate, conflicting, or truncated evidence
  blocks a complete object-coverage claim.
- Runtime guidance now routes AI operators to this status before describing
  backup state to a human.

## Why It Matters

WOM treats local reviewed artifacts as canonical and external systems as
backup or replica layers. That model only protects data sovereignty if the AI
also refuses to turn configuration or hopeful labels into fake completion.

The new status surface gives the AI counts and fixed states it can explain in
plain language. It also makes the missing evidence visible without uploading,
restoring, or asking the human to remember which remote operation happened.

## Safety Boundary

- A local commit is not remote GitHub proof.
- `declared_uploaded` is not proof that object bytes were verified.
- A generated index or database label is not external-database backup proof.
- Even full object receipt coverage does not prove that remote bytes still
  exist now.
- The command reads no objet bytes or zet bodies and calls no network,
  provider, database, model, or credential store.
- Output omits object ids, receipt paths, provider/store labels, local absolute
  paths, source content, and secrets.
- It writes nothing and never changes local authority.

## Verification

Regression coverage proves no-write baseline reporting, dry-run enforcement,
partial and full recorded-time object coverage through real execution-receipt
fixtures, forged missing-receipt rejection, bounded-scan truncation, output
redaction, and capability discovery. Release gates also cover the full test
suite, runtime skill validation, package resources, clean-wheel installation,
public privacy, and release readiness.
