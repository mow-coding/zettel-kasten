# v0.3.258 - Bounded Object-Storage Transport

v0.3.258 makes WOM-kit's default S3-compatible data path constant-memory with
respect to object size and tightens the remote-evidence boundary discovered
during the same independent audit. Single PUTs no longer materialize a path up
to the 5 GiB multipart threshold, and verification GETs no longer retain the
downloaded object.

## What Changed

- A single path PUT now uses a replayable iterable that reopens the exact path
  and yields at most 1 MiB per read. It preserves the explicit signed
  `Content-Length`, `Content-Type`, and real SigV4 payload SHA-256; it fails if
  the path emits more or fewer bytes than the signed length.
- A successful whole-object GET is hashed while streaming. The default sender
  returns only lowercase SHA-256, byte count, completeness, and length-proof
  scalars; it does not return the object body.
- Multipart XML, redirect, and provider-error bodies are retained only up to a
  64 KiB prefix. A declared `Content-Length` must be strict ASCII decimal and
  match the bytes read before authority-bearing control XML is trusted.
- Automatic redirects are disabled so SigV4 `Authorization` and `x-amz-*`
  headers cannot be forwarded to another origin.
- HEAD and whole-object GET evidence require HTTP 200; only HEAD 404 proves absence.
  Network, auth, rate, redirect, partial, invalid-length, truncated,
  and contradictory responses become unavailable evidence; they do not
  authorize an implicit PUT, adopt, skip, or conflict claim.
- `CompleteMultipartUpload` now detects the provider's HTTP-200-with-`<Error>`
  response form, classifies it as failure, and aborts the in-progress upload
  before bounded retry.
- The executor no longer issues an unconditional DELETE after a mismatch. A
  complete GET proves the generation read, but a correct concurrent replacement
  may occupy the key before DELETE. The remote object is preserved until a
  future provider-supported generation/ETag-conditional cleanup contract.

## Compatibility And Migration

- No archive, manifest, receipt, or provider schema migration is required.
- The injected sender keeps the exact five keyword arguments: `method`, `url`,
  `headers`, `data_path`, and `data_bytes`. Existing test and integration senders
  that return a complete `body: bytes` remain supported when length proof is
  available.
- Multipart part sizing, the default 5 GiB threshold, retry and cumulative PUT
  ceilings, credential references, key strategies, tier gates, and public CLI
  arguments are unchanged.
- A run that previously overwrote or adopted after an ambiguous HEAD/GET now
  fails closed. An explicit reviewed `--force-reupload` remains the deliberate
  override; ordinary repair does not treat provider unavailability as absence.

## Verify The Upgrade

```powershell
archive version <project-or-archive-root> --format json

archive object-storage-upload-plan <archive-root> `
  --only sha256:<object-digest> `
  --max-objects 1 `
  --dry-run `
  --format json

archive object-storage-upload-verify <archive-root> `
  --only sha256:<object-digest> `
  --max-objects 1 `
  --dry-run `
  --format json
```

Keep the first live provider operation tiny and follow the existing reviewed
tier gate. A failed verification may deliberately leave a correct remote object
present; investigate the provider response before choosing any cleanup action.

## Privacy And Safety Boundary

- Object bytes, request headers, authorization values, provider URLs, bucket
  names, raw error bodies, and local absolute paths are not added to durable
  results or receipts by this batch.
- Response-body canaries beyond the control cap are not retained. Failed or
  partial GET bytes never become a digest or mismatch claim.
- This release does not add provider-neutral conditional DELETE, automatic
  cleanup, a new AWS streaming-signature mode, a new dependency, or a live
  provider-validation claim.

## Provenance And Verification

The unbounded sender paths were reported by the independent Fable5 audit after
v0.3.257. Follow-up adversarial review reproduced additional fail-open cases in
verification GET, remote presence classification, redirects, multipart control
responses, and mismatch cleanup. Verification covers bounded replayable PUTs,
wire Content-Length without transfer chunking, streamed GET digest/count,
strict and oversized length fields, short/long request bodies, timeout and
partial response handling, HTTP 206, HEAD unavailability, presence-only adopt
and recorded-key revalidation, redirect refusal, capped control/error bodies,
short authority-bearing XML, CompleteMultipartUpload embedded errors and abort,
legacy fake senders, SigV4 vectors, the complete WOM-kit suite, packaged-resource
synchronization, release-readiness checks, clean-wheel installation, and public
release installation closure. The release supervisor must reinstall from the
published GitHub asset before final handoff. Real R2/AWS archive use remains a
separate beta-client confirmation step.
