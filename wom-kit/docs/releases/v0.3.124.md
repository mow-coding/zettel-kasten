# v0.3.124 - Notion Nested Tree Recovery Plan

Date: 2026-06-21

## What Changed

Real WOM archive migration work found a final gap after database-row import and
`contains` edge planning:

```text
Notion pages can contain deeper child pages whose parent is another page,
not a database row.
```

v0.3.124 adds a read-only recovery planner for that gap:

```powershell
archive notion-nested-tree-plan <archive-root> `
  --tree workbench/notion-nested-tree.sample.json `
  --source notion `
  --dry-run `
  --format json
```

MCP also exposes:

```text
notion_nested_tree_plan
```

## Recovery Model

The planner reads only a sanitized tree fixture and reports leaf pages as:

- `recovery_queue`: live content leaves that are missing and trace to a known
  generation root,
- `hold_queue`: leaves whose parent chain is missing, ambiguous, or still needs
  content review,
- `structure_skip_queue`: structure/template/view-container leaves that should
  not be minted as canonical zets by default.

The important behavior is negative as much as positive: when a parent record is
missing, WOM-kit reports `untraceable` instead of assigning the page to a
generation by guesswork.

## Safety Boundary

This release does not:

- call Notion or other providers,
- start OAuth,
- read real source exports,
- read page titles,
- read page bodies,
- read comments,
- download media,
- write zets,
- mint pages,
- write edges,
- write receipts,
- update object manifests,
- infer content/structure classification from private titles.

It only adds the fixture contract, parent-chain generation assignment, content
vs structure/template review queues, untraceable reporting, MCP/CLI exposure,
docs, and tests.
