# v0.3.214 - large-command progress and bounded output

v0.3.214 keeps long `ai-start-here`, `upgrade-check`, and CLI `zet-catalog`
runs visibly alive without exposing private archive content. It is additive and
requires no archive migration.

## What Changed

- Added opt-in `--progress` to all three commands.
- Added immediate stage boundaries, safe counted milestones, elapsed time,
  warming-up ETA, and 10-second quiet-interval heartbeats on stderr.
- Reduced path-bearing Doctor messages to content-free stage/count events.
- Added optional `--output .wom-scratch/diagnostics/<new-name>.json` for a full
  JSON result plus compact stdout summary.
- Confined output to JSON inside the private scratch surface and refused path
  traversal, non-JSON filenames, and overwrite.
- Added catalog path-metadata and frontmatter scan progress.

## Honest Boundary

The scratch result is not a zet, archive record, or receipt. It can contain
private result metadata, must not be committed, and should be deleted after
review. Core inspection writes nothing unless `--output` is explicitly used.

Catalog progress makes each scan visible. It does not add a persistent cache or
remove the full selected-scope scan performed by a new CLI process.

## Compatibility And Safety

Existing invocations that omit `--progress` and `--output` behave as before.
No archive schema changes, zets, manifests, receipts, indexes, providers, or
external backups are modified.
