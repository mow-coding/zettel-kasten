# v0.3.122 - AI Usage Observability Baseline

Date: 2026-06-21

## What Changed

WOM already had ways to tell an AI runtime where to start reading:
`runtime-context`, `operational_context`, overview-first zettel reads, and
scoped validation. v0.3.122 adds the first usage accounting layer so operators
can also ask:

```text
How much context did this AI task plan to read?
How many tokens did the runtime report using?
Which runtime, model, or purpose is becoming the bottleneck?
```

This is a local observability baseline, not live provider integration.

## Context Budget Planning

`archive ai-usage-plan --dry-run` estimates explicitly included
archive-relative UTF-8 text files against a token budget.

Example:

```powershell
archive ai-usage-plan <archive-root> `
  --budget-tokens 8000 `
  --include archive.yml `
  --include ops/operational-context.yml `
  --task-id task:next-action `
  --purpose "next action planning" `
  --dry-run `
  --format json
```

The output includes paths, byte counts, character counts, SHA-256 hashes, and
estimated tokens. It does not echo file contents.

The planner blocks source/object byte-store paths such as `objects/sha256/`.

## Usage Receipts

`archive ai-usage-record --dry-run|--approve` records reviewed token counters
under:

```text
receipts/ai-usage/
```

The receipt can store:

- task id
- runtime label
- model label
- purpose label
- input tokens
- output tokens
- total tokens
- cached input tokens when reported
- reasoning tokens when reported
- optional budget and planned-token metadata

It stores no prompt body and no response body.

## Usage Reports

`archive ai-usage-report --dry-run` aggregates local AI usage receipts by:

- runtime
- model
- purpose

This gives WOM a first local answer to "where are the token bottlenecks?"
without requiring provider billing integration.

## Compatibility

No archive migration is required.

The new commands are additive:

```powershell
archive ai-usage-plan <archive-root> --budget-tokens 8000 --include archive.yml --dry-run --format json
archive ai-usage-record <archive-root> --task-id task:x --runtime codex --model gpt-5 --purpose planning --input-tokens 1000 --output-tokens 250 --approve --reviewed-by person:reviewer --format json
archive ai-usage-report <archive-root> --dry-run --format json
```

## Safety Boundary

This release does not:

- call LLM providers,
- retrieve live provider usage or billing,
- store prompts,
- store responses,
- read source object bytes,
- read secrets,
- write zets,
- enforce hard runtime budgets,
- add MCP write tools,
- add a worker, queue, daemon, or scheduler.

The receipts are deliberately small. They preserve usage counters and
non-secret labels so the archive can learn where AI work becomes expensive
without turning private conversations into public or durable prompt logs.
