# v0.3.259 - Completed-Result Terminal Semantics

v0.3.259 closes the remaining no-`--output` terminal edge around `index` and
`index-health`. Once either command has computed its result, terminal delivery
is advisory and cannot replace that result with an unhandled output exception.

## What Changed

- Completed no-`--output` JSON uses the existing best-effort terminal wrapper
  around the unchanged serializer.
- Completed no-`--output` text keeps the same lines, order, wording, and final
  newline while using the same best-effort transport boundary.
- A successful committed index and current health result keep exit code 0 when
  stdout or stderr is broken, closed, or cannot encode the output.
- A completed quarantining index and stale/incomplete health result keep exit
  code 1 under the same terminal failures.
- A failed standard stream is replaced by a discard-only sink, preventing
  CPython's interpreter-shutdown flush from changing the process code to 120.

## Compatibility And Migration

- No archive, zettel, index, result, receipt, or package-resource schema
  migration is required.
- Command names, arguments, normal JSON schemas, normal text content, progress
  output, and the explicit `--output` path are unchanged.
- Failures before a result is computed still fail normally. `KeyboardInterrupt`
  and `SystemExit` are not converted into completed results.

## Operational Boundary

Best-effort terminal delivery does not prove that a human or host observed the
message. For an unattended or long operation, continue to provide a reviewed
archive-relative path such as:

```powershell
archive index <archive-root> `
  --output .wom-scratch/diagnostics/index-result.json `
  --format json

archive index-health <archive-root> `
  --dry-run `
  --output .wom-scratch/diagnostics/index-health-result.json `
  --format json
```

The complete-only artifact remains the durable evidence. The process exit code
describes the computed command result; it does not attest to terminal delivery.

## Privacy And Safety Boundary

- Terminal transport exceptions are not echoed or stored.
- No new zettel path, title, frontmatter, body, provider value, absolute local
  path, credential reference, token, or secret is added to output.
- A terminal failure cannot relabel a completed nonzero result as success.
- This batch does not change MCP error sanitation, index mutation logic,
  result publication, object storage, or approval workflows.

## Provenance And Verification

The no-`--output` edge was retained from the independent Fable5 audit after
the four P2 findings had already been released in v0.3.256 and v0.3.258. A
red-first regression reproduced a committed healthy index followed by an
escaping `BrokenPipeError`. A later actual-process review found CPython's
shutdown re-flush still changed the OS return code to 120, so release was held
again. Focused verification now closes real child-process stdout and progress
stderr in addition to covering success/current JSON, quarantined/stale text,
durable-output survival, post-commit progress failure, sanitized no-output
failures, and normal output compatibility. The final independent re-review
exercised real success, quarantine, stale/incomplete, progress, shared-stream,
JSON, and text subprocess cases and found no P0/P1/P2.

The complete local WOM-kit suite passed 1,517 tests with 13 skips and 4,226
passing subtests. Public release gates and package-resource synchronization also
passed. Publication closure additionally requires a clean-wheel install and a
fresh installation from the published GitHub asset before final handoff.
