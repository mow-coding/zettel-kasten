# v0.3.112 - Receipt SHA Evolution And Zettel Batch Scale

Date: 2026-06-19

## What Changed

This release handles the tenth real-use feedback sweep after v0.3.111.

Three areas changed:

- mint and retired-draft target SHA checks now understand approved edge-only
  canonical zet evolution;
- mint snapshots preserve original draft bytes, and legacy LF/CRLF-only
  source/snapshot pairs can still retire;
- `zettel-edge-batch` now preloads a zettel id/path index for zet-to-zet batch
  plans.

## Receipt SHA Evolution

`archive validate` and `archive doctor` still verify mint and retired-draft
receipt SHA-256 references.

When a canonical zet target SHA differs, validation now checks whether the
current canonical file can be reconstructed by removing only approved
post-receipt `zettel-edge` writes from frontmatter `edges` and restoring the
historical `updated_at` candidate.

If that exact reconstruction matches the historical receipt SHA, validation
reports an informational diagnostic instead of a SHA mismatch error.

Body edits, non-edge frontmatter edits, missing edge receipts, or unapproved
changes still remain integrity errors.

## Snapshot Newline Safety

New mint snapshots are written from the original draft bytes, so Windows text
mode no longer changes LF-only drafts into CRLF snapshots.

For older archives, `archive retire-draft` can continue when the mint receipt
source and snapshot hashes differ only by LF/CRLF newline normalization. The
command returns a warning rather than treating that legacy pair as impossible to
retire.

## Batch Zettel Target Scale

`archive zettel-edge-batch` already preloaded the object manifest for objet
targets.

It now also preloads one zettel id/path index for policy-writable batch rows
that use `from_zettel` or direct `zet_<id>` targets. This avoids repeated full
archive scans for zet-to-zet batch plans while keeping every item routed through
the same single-edge gate.

## Safety Boundary

This release does not:

- rewrite existing mint receipts,
- rewrite existing retired-draft receipts,
- add an automatic re-seal command,
- accept arbitrary target SHA drift,
- infer new edge meanings,
- read source exports,
- read object bytes,
- call providers,
- start OAuth,
- add MCP write tools.
