# v0.3.189 - Doctor mint receipt internal liveness

v0.3.189 makes `doctor --strict --progress` more useful when a large archive
slows down inside a mint receipt after the receipt path has already been printed.

The previous release proved receipt-level heartbeat mode works, but a slow
individual receipt could still look silent because non-sampled receipts did not
print their internal file-reference steps.

## Every-Receipt Internal Liveness

Mint receipt progress now emits these content-free liveness markers for every
receipt:

- `started receipt checks`
- `checking source file ref`
- `checking target file ref`
- `checking snapshot file ref`
- `completed receipt checks`

Detailed sub-steps are still sampled for the first four receipts, every 250th
receipt, and the final receipt. The heartbeat-mode transition message now says:

```text
continuing with receipt heartbeat; minimal file-ref liveness every receipt; detailed substeps every 250 receipts
```

This makes the next reported pause actionable: the operator can see whether the
current receipt is loading, checking a source/target/snapshot file reference, or
has already completed.

## Large-File Hash Liveness

When a mint receipt references a large file, doctor can now emit content-free
SHA-256 liveness while hashing:

```text
hashing target file bytes
still hashing target file bytes
hashed target file bytes
```

The messages identify only the receipt section being checked. They do not echo
file bodies, object ids, provider locations, local absolute paths, or secret
values.

## ETA Warm-Up

Counted progress now keeps `eta=warming_up` longer: through the first nine
samples or the first 30 seconds of a stage. This avoids early multi-hour
projections from a tiny sample.

## Compatibility

Additive; **no migration required**.

Only optional `--progress` stderr output changes. Result JSON, diagnostics,
receipts, manifests, and archive files are unchanged.
