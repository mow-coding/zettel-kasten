# v0.3.205 - host-AI exhaustive abstract reading

v0.3.205 connects the live local zet catalog to the MCP and runtime surfaces
used by host AI applications. It is additive and requires no archive migration.

## What Changed

- Added read-only MCP `zet_catalog` with canonical/draft/all scope,
  deterministic cursor pages, explicit completion coverage, and
  snapshot-change blocking.
- Added `section` to MCP `read_zettel`; `overview` gives the compact first read,
  while omitted `section` keeps the compatibility `body` default.
- Added optional bounded `abstract` to MCP `create_draft_zettel`.
- Added exhaustive catalog reading to runtime context, AI start-here, personal,
  company, and family archive templates, and the shipped runtime skill.
- Host instructions now require every catalog page under one snapshot before
  an archive-wide coverage claim, then use abstracts, ties, and edges to choose
  body-reading order.

## Compatibility

Existing archives need no migration. Existing MCP callers that omit `section`
continue receiving body reads. `list_zettels` remains available for
compatibility, but it is not an exhaustive coverage contract.

## Safety And Scope

The catalog is dry-run only. It reads local frontmatter, reads and echoes no zet
body text, needs no generated index, reads no objet bytes or secrets, calls no
providers, and writes no files.

Goal, loop, task branching, and completion UI remain in the host LLM
application. WOM supplies local memory and completion evidence; it does not
record traversal state or make a generated map authoritative.
