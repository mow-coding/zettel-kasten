# v0.3.231 - First-Read Readiness Gate

v0.3.231 separates ordinary archive health from whether a newly entering AI
can reconstruct local memory through an explicit compact first read for every
canonical zet. It is additive, read-only, and requires no archive migration.

## What Changed

- New CLI command `archive first-read-readiness <archive-root> --dry-run` with
  aliases `zet-first-readiness` and `memory-readiness`.
- New MCP tool `first_read_readiness` with the same frontmatter-only contract.
- The gate distinguishes explicit abstracts, compatibility first reads,
  missing text, unreadable frontmatter, redaction, duplicate ids, and
  unaddressable ids.
- `state: ready` requires explicit abstracts for every non-redacted canonical
  zet and one uniquely resolvable safe id for every selected entry.
- `status-board` now reports first-read attention counts and bounded
  content-free candidates.
- AI start-here and archive templates now place this gate before the complete
  private `zet-catalog-pass` workflow. Quick start still does not scan zets.

## Upgrade And Compatibility

No migration or automatic backfill is performed. Existing `gist`, `summary`,
`description`, or `overview` values remain usable compatibility first reads,
but they produce `state: compatibility_only` until an explicit abstract is
human-reviewed and approved.

Run:

```text
archive first-read-readiness <archive-root> --dry-run --progress --format json
```

A nonzero exit with a structured `needs_attention` result means the gate found
work; it does not mean the scan crashed.

## Safety And Privacy

- Reads canonical frontmatter only; no zet body or objet byte is read.
- Returns no title, abstract, or body text and no duplicate-id values.
- Calls no provider, network, model, or credential store and writes nothing.
- Does not judge semantic quality or prove that a host model consumed every
  first read.
- Missing abstracts remain a human-reviewed repair queue; no write approval is
  inferred from this result.
