# v0.3.145 - zet Quality And Archive-Root Boundary Checks

v0.3.145 responds to real long-project feedback where canonical zets, OCR
tables, raw notes, strategy notes, correction events, and external reports can
blur together.

## What Changed

- Added read-only `archive zet-quality-check`, with alias
  `archive zettel-quality-check`.
- Added optional `zet-quality-rules.yml` project entity rules. A forbidden
  entity alias rule can become a blocker-severity quality issue without
  echoing the matched term.
- `mint-zet --dry-run` now includes the same `quality_check` summary and blocks
  only when quality issues are blocker-severity.
- The quality check reports risks for:
  - entity-term rules,
  - document type,
  - audience,
  - OCR/parse metadata in canonical body,
  - table structure review and row mapping,
  - correction events,
  - source acquisition, verification, and rights metadata,
  - derived artifact source/sync metadata,
  - internal working-context residue in external-facing zets.
- `archive doctor` now warns when the archive root contains top-level web/app
  development artifacts or an incomplete `.git` marker.
- Generated archive `.gitignore` defaults and artifact hygiene policy now
  include `node_modules/`, `.next/`, and `.vercel/`.

```powershell
  --format json
```

## Safety Boundary

The new quality gate:

- reads one selected zet,
- writes nothing,
- calls no providers,
- creates no revision candidate,
- syncs no external artifact,
- performs no audience-aware rendering,
- echoes no matched entity terms, zet body text, frontmatter values,
  source-ref values, provider URLs, local absolute paths, tokens, or secret
  values.
