# v0.3.240 - Scalable First-Read Diagnostics

v0.3.240 turns one successful but slow real-archive validation into a bounded
diagnostic improvement. It reduces abstract-freshness receipt work, makes its
two stages visible, clarifies first-read exit semantics, and defines a small
human-reviewed abstract pilot.

## What Changed

- `abstract-freshness` scans canonical zets first and builds evidence targets
  only for current explicit abstracts.
- Safe zet ids directly address their deterministic mint/promotion receipts.
  Canonical-revision and abstract-backfill evidence stay bounded to their own
  receipt directories.
- An explicit-abstract target without a safe id conservatively falls back to
  bounded publication-directory scans.
- The command reports candidate lookup mode, target count, no whole-tree
  enumeration, and no persistent cache use.
- Progress identifies `abstract-canonical` as `stage=1/2` and
  `abstract-evidence` as `stage=2/2`; stage one names stage two before moving
  on.
- First-read result schema v0.2 uses `ok` for safe diagnostic completion and
  `readiness_met` for the readiness verdict. Completed attention states exit
  zero; blocked input and execution failures do not.
- English and Korean three-zet pilot guides route legacy missing abstracts
  through existing private proposal, human review, approval, revert, and audit
  contracts without authorizing bulk automatic writing.

## Upgrade And Compatibility

No archive migration is required. Restart the AI operator process.

Automation that previously treated `first-read-readiness` process exit 1 as a
normal attention queue should now inspect `state`, `readiness_met`, or
`readiness.first_read_surface_ready`. Process exit zero proves only that the
diagnostic completed safely.

The abstract-freshness result remains schema v0.1 and retains its freshness
classifications. Scan metadata is additive. The command creates no cache file
and writes no archive state.

## Safety And Claim Boundary

- Candidate routing never suppresses evidence for a safe target; unsafe ids
  select a conservative fallback.
- No title, abstract/body text, hash value, receipt path, reviewer id, provider
  URL, absolute path, or secret is added to output.
- The three-zet pilot requires separate human review and explicit approval.
- A fresh hash pair is still not proof of truth, usefulness, completeness, or
  model consumption.

## Verification

- Focused tests cover candidate-only receipt reads, safe fallback behavior,
  two-stage progress labels, v0.2 exit semantics, text output, MCP parity, and
  existing mint/promotion/backfill/revision evidence recognition.
- Final repository suite: `1379 passed, 11 skipped`.
- Public privacy and release-readiness publication gates passed before the
  tag and GitHub release were created.
