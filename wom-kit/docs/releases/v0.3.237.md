# v0.3.237 - Canonical Revision Restore Plan

v0.3.237 adds a read-only plan for connecting separately recovered old zet
bytes to one valid canonical revision receipt.

## What Changed

- Added CLI `zet-revision-restore-plan` with aliases
  `canonical-revision-restore-plan` and `zet-restore-plan`.
- Required a healthy archive-wide canonical revision receipt/lock audit before
  selected-receipt planning.
- Required the current canonical zet to match the receipt's full `after` state
  and the recovered private full zet to match its full `before` state.
- Rechecked the historical bytes against current schema, explicit-abstract,
  kind, object-reference, edge, private-locator, quality, and self-containment
  policy.
- Bound archive audit, receipt, current/recovered states, restore delta, and
  policy results into one plan digest.
- Kept output content-free and the operation local, provider-free, and
  write-free. No MCP tool was added.

## Upgrade And Compatibility

No archive migration is required. Restart the AI operator process so the new
CLI command is visible. Recover old full-zet bytes from a trusted private
backup into `.wom-scratch/revisions/restores/`; never commit that scratch file.

## Safety And Claim Boundary

- Receipt hashes do not recreate old content; full old bytes are mandatory.
- A selected receipt cannot bypass a broken or ambiguous archive-wide history.
- A later revision, altered restore file, changed receipt, changed project
  policy, or unresolved target lock blocks or changes the plan.
- Output omits zet identity/path, receipt path, proposal filename, reviewer,
  title, abstract/body text, custom frontmatter, provider URL, absolute path,
  and secrets.
- `ready_for_human_review` authorizes no write. No restore writer exists in
  this release, and manual canonical copying remains prohibited.
