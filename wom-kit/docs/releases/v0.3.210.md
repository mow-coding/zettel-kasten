# v0.3.210 - explainable connection order

v0.3.210 lets a host explain why a seeded connection walk placed each zet in
its reading order, without making that extra payload the default. It is additive
and requires no archive migration.

## What Changed

- Added CLI/MCP `projection=routed_reading`, valid only with
  `order=seeded_connection_walk` and verified start zet ids.
- Added compact per-item seed, connection-passage, and disconnected-component
  route reasons with distance and stored-edge direction evidence.
- Kept `projection=reading` compact and free of per-item route metadata.
- Changed strict chain page identity from returned zet ids to opaque,
  snapshot-bound file-entry identities so duplicate ids remain distinct.
- Added `page_entry_identity_sha256` and explicit continuation-contract safety
  labels without returning entry identities or archive paths.
- Cached entry identities and order-specific workload estimates in the bounded,
  process-local catalog session.

## Scale Evidence

On the same temporary 10,000-node seeded chain and 8,000-token item budget:

- `reading`: 179 pages, 1,414,699 estimated item tokens, 50.8 seconds;
- `routed_reading`: 257 pages, 2,026,799 estimated item tokens, 53.8 seconds.

Both returned every node, reached node/abstract/follow-up readiness, and had no
over-budget page. These are local synthetic observations, not an SLA or
provider-reported token usage.

## Compatibility And Safety

Catalog schema advances to v0.6. Continuation schema advances to v0.3; restart
v0.3.209 in-flight strict passes at cursor zero. Existing archives and normal
reading calls need no migration.

No body, objet byte, secret, provider, model, generated index, or persisted
route/goal/loop state is read or created. Stored edge direction and meaning are
never changed.
