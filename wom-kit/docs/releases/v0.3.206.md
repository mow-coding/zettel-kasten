# v0.3.206 - zet-catalog scale and token budget

v0.3.206 makes exhaustive abstract reading measurable and practical across
host-application loops. It is additive and requires no archive migration.

## What Changed

- Added scope/page abstract characters, items-only JSON characters/bytes, and
  estimated token workload to CLI and MCP catalog results.
- Added optional CLI `--max-estimated-tokens` and MCP
  `max_estimated_tokens`; pages always return at least one item.
- Added a process-local MCP materialized snapshot for intermediate pages and a
  full local metadata revalidation before the completing page.
- Added explicit session-consistency and scan/cache evidence fields.
- Accelerated safe YAML reads with C safe loader when available and fallback to
  the prior safe loader; cold catalog parsing uses a bounded thread pool.
- Added `tools/benchmark_zet_catalog.py`, a subprocess smoke test, and a
  1,000-node exhaustive coverage/cache regression.

## Scale Evidence

A synthetic 10,000-zet run with 120-character abstracts and an 8,000-token
page budget completed 264 pages in 54.6 seconds on the development Windows
host. It collected 10,000 unique ids, parsed frontmatter 10,000 times, checked
path metadata 20,000 times, and returned no over-budget page (largest estimate
7,848). These are reproducible local observations, not an SLA or provider token
report.

The same fixture estimated 300,000 tokens for abstract text alone and 2,064,699
tokens for full catalog items. That gap is why the host must inspect workload,
continue across loops, and never confuse one context window with full coverage.

## Safety And Compatibility

Existing archives and callers need no migration. Omitting the token budget
preserves item-count pagination. The cache is process-local, bounded,
non-persistent, non-authoritative, and writes no map or traversal state.

The catalog still reads no zet bodies, objet bytes, providers, or secrets and
requires no generated index.
