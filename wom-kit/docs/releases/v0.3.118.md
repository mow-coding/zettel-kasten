# v0.3.118 - Mint Index Staleness Fast Path

Date: 2026-06-20

## What Changed

This release closes the remaining large-batch mint scale gap after v0.3.114 and
v0.3.116.

v0.3.114 moved mint duplicate comparison onto `db/archive-index.sqlite` when the
generated index is current. v0.3.116 then avoided archive-wide id scanning for
standard `--zettel-id` mint source resolution.

Real-use profiling found one more O(n) step: before using the generated index,
mint still checked staleness by globbing every canonical zettel and statting each
file mtime. In large archives that made every mint grow with canonical archive
size.

v0.3.118 adds `index_metadata` to the generated SQLite index. `archive index`
now records canonical zettel count, canonical max mtime, update time, and the
WOM-kit version that refreshed the metadata. Generated-index-backed `mint-zet`
uses that metadata instead of a live canonical glob/stat pass.

The same checkpoint hardens generated-index SQLite access for large batches:
archive index connections now use a 30s `busy_timeout`, write paths use WAL
mode, and WAL/SHM/journal sidecars are treated as rebuildable generated
artifacts in `.gitignore` repair and artifact hygiene checks.

## Batch Impact

After rebuilding the index once with v0.3.118 or later:

```powershell
archive index <archive-root> --format json
```

large mint loops can keep the fast path:

- duplicate comparison reads generated index rows,
- staleness validation reads `index_metadata`,
- approved mint upserts the new canonical row,
- approved mint also updates `index_metadata`.
- index/mint generated-index writes wait on short SQLite locks and use WAL mode.

The `duplicate_check` JSON now reports:

```text
staleness_check: index_metadata
live_staleness_paths_checked: 0
```

## Compatibility

Older generated indexes do not have `index_metadata`. WOM-kit does not trust
those blindly. It falls back to the legacy live staleness scan until the user
rebuilds the index with:

```powershell
archive index <archive-root> --format json
```

## Safety Boundary

This release does not:

- remove `index-health`,
- claim that external manual zettel edits are impossible,
- auto-rebuild the index during mint dry-run,
- change duplicate semantics,
- rewrite zettel bodies,
- remove inbox drafts,
- call providers,
- add UI behavior,
- add MCP write tools.
- add a background worker, daemon queue, or multi-process scheduler.

If a human or another tool edits canonical zets outside WOM-kit during a batch,
run `archive index-health --dry-run` or rebuild the index before continuing the
large mint run.
