# v0.3.190 - Doctor mint target file-ref drilldown

v0.3.190 drills further into `doctor --strict --progress` when a mint receipt is
slow inside `checking target file ref`.

The previous release named the current receipt and the source/target/snapshot
file-ref phase. This release names the inner steps of the file-reference check.

## File-Ref Drilldown

Mint receipt file-reference checks can now emit:

- `resolving target file path`
- `checking target file exists`
- `resolved target file ref <archive-relative-path>`
- `checking target sha256 field`
- `reading target file stat`
- `target file sha256 cache hit`
- `hashing target file bytes`
- `still hashing target file bytes`
- `hashed target file bytes`
- `checking target edge-receipt evolution`
- `target file sha256 mismatch`
- `target file ref ok`

The same pattern applies to source and snapshot refs.

## Hash Start Guarantee

When a mint receipt file ref needs a fresh SHA-256 read, doctor now emits
`hashing <section> file bytes` before reading the file, regardless of file size.

`still hashing ... file bytes` remains a chunk heartbeat for longer reads.
Small files usually show only:

```text
hashing target file bytes
hashed target file bytes
```

## Retired Source Clarity

When a mint source has already been closed by a retire-draft receipt, progress
now says:

```text
source file ref skipped; source retired
```

instead of silently omitting the source file-ref check.

## Compatibility

Additive; **no migration required**.

Only optional `--progress` stderr output changes. Result JSON, diagnostics,
receipts, manifests, and archive files are unchanged.
