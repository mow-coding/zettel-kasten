# v0.3.188 - Doctor mint receipt progress heartbeat

v0.3.188 tightens `doctor --strict --progress` for large archives with many
mint receipts.

The previous checkpoint made the first three mint receipts very visible. That
proved the early receipt checks were completing, but it still left a confusing
gap immediately after the third receipt: operators could not tell whether doctor
had advanced to the fourth receipt, entered another global step, or stopped.

This release closes that diagnostic gap.

## Mint Receipt Heartbeat

`doctor --strict --progress` now prints a mint-receipt entry for every checked
mint receipt.

The detailed internal sub-steps remain sampled, so progress output stays usable
on large archives:

- detailed sub-steps for receipts 1 through 4,
- detailed sub-steps every 250 receipts after that,
- detailed sub-steps for the final receipt,
- a receipt-level heartbeat for every receipt.

After the fourth receipt, doctor also emits:

```text
continuing with receipt heartbeat; detailed substeps every 250 receipts
```

That line tells the operator the tool intentionally moved from dense diagnostic
sampling to lighter heartbeat mode.

## ETA Warm-Up

Progress ETA now uses per-stage timing instead of whole-command timing.

For the first few samples of a counted stage, ETA is reported as:

```text
eta=warming_up
```

This avoids projecting a misleading multi-day ETA from the first one to four
items of a large stage.

## Compatibility

Additive; **no migration required**.

Only optional `--progress` stderr output changes. Result JSON, archive files,
receipts, manifests, and validation semantics are unchanged.
