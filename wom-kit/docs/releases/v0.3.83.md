# v0.3.83 - Derived Text Manifest Quality Gate

Date: 2026-06-17

Status: v0.3.83 read-only derived text manifest quality checkpoint

v0.3.83 continues the client feedback #9 path by making `derive-text coverage`
check not only whether textual objets have derived-text rows, but also whether
those rows carry enough extraction provenance to be auditable.

## Added

- `manifest_quality` in `archive derive-text coverage --dry-run`.
- The same output through alias `archive derive-text-coverage --dry-run`.
- Coverage blocker `derived_text_manifest_quality_issues`.
- Documentation for `tool_version` quality expectations.

## Behavior

The coverage gate now checks existing `objects/manifests/derived-text.jsonl`
records for:

```text
source_object_id
derivation_kind
tool_name
tool_version
review_status
```

`tool_version` must identify the extractor, parser, OCR, ASR, model, or script
version that produced the derived text. Blank values and weak labels such as
`unknown`, `n/a`, `none`, `todo`, or `tbd` are treated as quality issues.

If textual coverage is numerically complete but manifest quality is not, the
gate returns `ok: false` so an AI runtime cannot claim the extraction pass is
complete.

## Safety Boundary

This release remains read-only. It does not read source bytes, read
derived-text bodies, run parsers, run OCR, run ASR, call LLM vision, call
providers, write derived text, write receipts, draft zets, mint zets, echo
local paths, echo tool paths, echo provider URLs, or echo secret values.
