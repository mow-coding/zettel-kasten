# v0.3.199 - Object-storage WOM location reconcile

v0.3.199 follows the v0.3.198 reconcile status release by closing the next
object-storage receipt/manifest binding gap.

No archive migration is required.

## What Changed

`archive doctor` now distinguishes two cases that previously looked the same:

1. A genuine missing `wom_uploaded` manifest binding after a successful
   object-storage execution receipt.
2. A repeated `skipped_remote_same` execution receipt for the same object,
   provider, store, and remote key already covered by an existing valid
   `wom_uploaded` location.

The second case is no longer reported as
`object_storage_upload_wom_location_missing`. The manifest writer de-duplicates
one `wom_uploaded` location per remote key, so a second exact
`execution_receipt_ref` would be impossible and unnecessary.

For the first case, doctor now points to a dry-run-first repair command:

```text
archive object-storage-wom-location-reconcile <archive-root> \
  --receipt receipts/providers/object-storage-executions/<receipt>.json \
  --dry-run
```

After review, the operator can approve the local manifest binding repair:

```text
archive object-storage-wom-location-reconcile <archive-root> \
  --receipt receipts/providers/object-storage-executions/<receipt>.json \
  --approve \
  --reviewed-by <actor>
```

## Safety

- The command requires exactly one of `--dry-run` or `--approve`.
- `--approve` requires `--reviewed-by`.
- `--receipt` must be an archive-relative object-storage execution receipt path.
- `--max-items` can bound batch repair plans.
- The command never calls providers.
- The command never reads credentials.
- The command never reads object bytes.
- The command never uploads, downloads, syncs, or checks remote availability.
- Candidate output does not echo object ids, remote keys, bucket names, provider
  URLs, exact credential refs, or local absolute paths.
- Approved mode writes only the object manifest plus one audit receipt under
  `receipts/providers/object-storage-manifest-reconciles/`.
