# v0.3.255 - Crash-Safe Index Rebuild And Durable Result Capture

v0.3.255 turns real-use feedback from a large WOM archive into a safer generated
index and a durable way to judge long-running CLI commands.

## What Changed

- `archive index` now rebuilds the complete generated index inside one explicit
  SQLite transaction. Schema work, old-row deletion, new rows, and metadata
  commit together; interruption before commit preserves the prior index.
- `archive index` and `archive index-health` add opt-in `--progress` with fixed,
  content-free stage/count messages and a 10-second heartbeat on stderr.
- Both commands add opt-in `--output
  .wom-scratch/diagnostics/<new-name>.json`. The complete result and scoped exit
  code are published atomically without overwriting an existing file.
- A first-build SQLite file with no committed `zettels` table is reported as
  `archive_index_schema_incomplete` / `stale_or_incomplete` instead of a raw SQL
  exception.
- Invalid zet or view YAML, terminal pipe loss, concurrent result publication,
  symbolic links, and Windows junctions stay inside a sanitized failure
  boundary without publishing parser excerpts or local absolute paths.

## Who Should Upgrade

Upgrade if an archive uses `archive index` or `index-health`, especially when
the archive is large enough for commands to outlive a terminal or AI execution
window. v0.3.254 users who observed missing final output or an empty generated
index should upgrade before rebuilding again.

## Compatibility And Migration

- No archive schema or zet migration is required.
- Existing zets, objets, manifests, receipts, views, source maps, and generated
  indexes remain compatible.
- The index is still disposable generated state. `index-health` remains
  read-only and never starts an automatic rebuild.
- `--progress` and `--output` are optional. Without them, existing stdout and
  exit-code behavior remains compatible.

## Verify The Upgrade

Use a new result filename for every attempt:

```powershell
archive version <project-or-archive-root> --format json

archive index-health <archive-root> `
  --dry-run `
  --progress `
  --output .wom-scratch/diagnostics/index-health-before.json `
  --format json
```

If health reports a missing or stale index, run one explicit rebuild and then a
fresh health check:

```powershell
archive index <archive-root> `
  --progress `
  --output .wom-scratch/diagnostics/index-rebuild.json `
  --format json

archive index-health <archive-root> `
  --dry-run `
  --progress `
  --output .wom-scratch/diagnostics/index-health-after.json `
  --format json
```

The final confirmation requires `cli_execution.exit_code: 0`,
`index_state: current`, and matching live/indexed zettel counts. A missing output
file after interruption is an unknown result, never success evidence.

## Privacy And Safety Boundary

- Progress does not contain zet ids, titles, body text, object bytes, provider
  values, local absolute paths, tokens, or secrets.
- Result files are private local scratch, not receipts, canonical zets, or files
  to commit.
- A completed result records the command result before later terminal delivery;
  it does not claim that an AI wrapper displayed the summary.
- SQLite commit and scratch publication are separate. If result publication
  fails after commit, run a new read-only health check before deciding whether
  another rebuild is needed.

## Provenance And Verification

The change was prompted by basoon v0.3.254 real-use feedback after a successful
mint was followed by missing command output and a later confirmed
8,586-live/0-indexed state. The implementation received independent release,
privacy, transaction, Windows-path, concurrency, and error-redaction reviews.
Focused regressions, the complete WOM-kit suite, package-resource synchronization,
release-readiness checks, and clean-wheel installation are required for
publication. Real-use confirmation remains a separate beta-client step.
