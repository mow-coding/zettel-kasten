# v0.3.4 - Derived Text Capture

Release date: 2026-06-13

v0.3.4 adds the first implemented derived text layer for source objets. This is
the safe registration path for text that has already been extracted by an
external parser, OCR tool, ASR tool, or LLM vision workflow.

## Added

- CLI `archive derive-text capture <archive-root> --text-file <file> --source-object-id <object-id> --derivation-kind <kind> --tool-name <name> --tool-version <version> --review-status <status> --dry-run|--approve`.
- Derived text manifest records in `objects/manifests/derived-text.jsonl`.
- Local derived text body storage under `objects/derived-text/sha256/`.
- Approved capture receipts under `receipts/derived-text-capture/`.
- Doctor/schema validation for derived text records, source object references,
  vocabulary, and stored text hashes.
- `archive index` ingestion and `archive search` results with
  `type: derived_text`.

## Vocabulary

First implemented `derivation_kind` values:

- `parser`
- `ocr`
- `asr`
- `llm_vision`

First implemented `review_status` values:

- `unreviewed`
- `human_corrected`

## Compatibility

- Existing v0.3.3 archives do not need a frontmatter migration.
- Existing archives can omit `objects/manifests/derived-text.jsonl`; missing
  derived text manifests are treated as zero derived text records.
- The source object for a derived text record must already exist in
  `objects/manifests/files.jsonl`.
- Rebuild the generated search index after approved derived text capture:

```text
archive index <archive-root>
```

## Not Included

This release does not run OCR, ASR, parser extraction, LLM vision, provider APIs,
drafting, minting, Notion sync, object upload/sync, ZET transport, or MCP write
tools. It only registers already extracted UTF-8 text after explicit approval.
