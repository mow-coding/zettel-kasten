# v0.3.196 - Doctor progress-log path-policy clarification

v0.3.196 follows the v0.3.195 doctor output path-policy release with the
matching clarification for `doctor --progress-log`.

No archive migration is required.

## What Changed

`archive doctor --summary --progress-log <path>` and summary-producing doctor
runs now include a `progress_log` block when a progress log path is requested.

For a relative progress-log path, the summary reports:

```json
{
  "written": true,
  "path": "logs/doctor-progress.jsonl",
  "path_kind": "cwd_relative",
  "relative_to": "current_working_directory",
  "absolute_path_echoed": false,
  "contains": "progress_events_jsonl",
  "tracking_policy": "local_progress_artifact_not_archive_receipt",
  "git_guidance": "Do not commit by default; keep locally, summarize into feedback, or delete after review."
}
```

This makes the intended difference explicit:

- `doctor --output logs/result.json` is archive-relative and writes under the
  archive root.
- `doctor --progress-log logs/progress.jsonl` is cwd-relative and writes under
  the process current working directory.

Absolute progress-log inputs are redacted in summaries as
`<absolute-progress-log-path>` while still reporting
`path_kind: absolute_input_redacted`.

## Operator Notes

- Full doctor result JSON files and progress JSONL files are local diagnostic
  artifacts, not archive receipts.
- Do not commit them by default.
- Keep them locally while reviewing a failing run, summarize the relevant
  counts/codes in feedback, or delete them after review.
- Default doctor diagnostics, progress-log write behavior, and exit-code
  semantics are unchanged.
