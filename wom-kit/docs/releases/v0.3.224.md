# v0.3.224 - Quick Runtime Context And No-Repeat AI Handoff

v0.3.224 closes the remaining automatic full-Doctor route in the AI entry
flow. It is additive and requires no archive migration.

## What Changed

- CLI and MCP runtime-context are quick by default and do not construct Doctor.
- CLI `--full-doctor --progress` and MCP `full_doctor: true` keep complete
  validation available through an explicit choice.
- Runtime-context reports inspection mode, Doctor status, and observed broad
  reads instead of leaving the read boundary implicit; the service refuses a
  full-Doctor result if read observations are omitted.
- `ai-start-here` marks runtime-context already included, adds
  `completed_commands`, `next_commands`, and `remaining_ai_runtime_order`, and
  suppresses the already-satisfied default runtime-context next step.
- Markdown start-here output separates `Already Included` from `Next Commands`.
- The bundled AI runtime skill starts with quick ai-start-here and tells the AI
  not to repeat runtime-context immediately.

## Upgrade Guidance

Normal AI entry remains:

```text
archive ai-start-here <archive-root> --dry-run --progress --format json
```

Raw runtime context is also quick:

```text
archive runtime-context <archive-root> --format json
```

Add `--full-doctor --progress` only when complete health validation is required.
MCP callers use `full_doctor: true` for the same explicit path. Scripts that
previously relied on runtime-context to run Doctor automatically must add that
option.

## Privacy And Work Boundary

Quick mode reads no zet body or objet byte and accesses no provider or
credential store. Full CLI progress remains content-free and path-redacted.
Start-here does not modify `ops/operational-context.yml`; it only avoids copying
an already-satisfied default instruction into its derived next-step list.
