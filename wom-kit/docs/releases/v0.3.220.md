# v0.3.220 - Receipt-Audited Deterministic Abstract Rollback

v0.3.220 adds the audited inverse of a successful v0.3.219 abstract backfill.
It is additive and requires no archive migration.

## What Changed

- Added `zet-abstract-backfill-revert` (`abstract-backfill-revert`).
- Requires the exact applied receipt SHA-256; the proposal file is not needed.
- Dry-run verifies source receipt schema/authority, current applied file/body/
  abstract hashes, and deterministic restoration of every before-file hash.
- A new approved revert requires a safe reviewer plus
  `--affirm-abstract-removal-reviewed`.
- Restores exact pre-backfill bytes by removing only the deterministic inserted
  `abstract:` line, while preserving the source receipt.
- Writes one immutable text-free revert receipt after all canonical writes pass.
- Restores the exact applied state when an in-process item or receipt operation
  fails.
- Validates a matching revert receipt/current state as no-write
  `already_reverted`.
- Keeps the short-lived advisory lock under private `.wom-scratch/`.

## Privacy And Authority

Public output omits source receipt path, reviewer, zet ids/paths, titles, bodies,
abstracts, and absolute paths. The private revert receipt retains target and
hash evidence but no body or abstract text.

A green audit proves exact reversibility only. It does not authorize removal,
and the AI must not infer the reviewer or affirmation.

## Honest Boundary

Any later canonical edit blocks rather than being merged or overwritten. The
lock does not cover external editors, and forced process or machine termination
can bypass in-process cleanup because no crash-recovery journal is written.

Reapplying the same text later requires a newly reviewed proposal byte sequence,
new proposal hash, and new applied receipt. Old evidence remains immutable.
